# TimeLens 产品迭代规范

> **目的**：把一期复盘中的有效做法固化为可重复流程，使后续版本在「产品设计 → 技术架构 → 任务排期 → 研发测试 → 运维交付」上步骤一致、责任清晰、可验收。  
> **适用**： major / minor 功能迭代与里程碑发布；紧急热修可裁剪部分环节但须书面记录裁剪项。  
> **版本**：v1.0 · 2026-03-30

---

## 1. 文档体系与角色

### 1.1 标准文档矩阵

| 文档类型 | 建议路径 | 维护责任 | 用途 |
|----------|----------|----------|------|
| 产品设计（PRD） | `prd/PRD_<阶段>_<主题>.md` | 产品 Owner | 范围、边界、验收叙事、与路线图对齐 |
| 技术架构 | `rule/TimeLens_TechArch_V*.md` | 技术 Owner | 分层、模块、数据模型、禁止项、性能与隐私 |
| 研发任务与验收 | `prd/<阶段>_研发任务与验收计划_P*.md` | 技术 Owner + 测试 | 模块依赖、Given/When/Then、排期参考 |
| 测试说明 | `docs/M*_验收与测试.md`（或按里程碑命名） | 测试 | 自动化映射、手动 E2E、性能模板、发布前 checklist |
| 用户向 | `docs/TimeLens_用户使用手册.md` 等 | 产品 / 研发 | 对外行为与已知限制一致 |
| 发布与打包 | `docs/DMG打包指南.md` + `scripts/*` | 研发 | 版本号、产物、发布命令 |

### 1.2 关键角色（可一人兼任）

| 角色 | 职责摘要 |
|------|----------|
| **产品 Owner** | 范围冻结与变更裁决、验收拍板、用户文档与 Release 叙事 |
| **技术 Owner** | 架构与实现边界、模块拆分、技术风险与降级策略 |
| **接口契约 Owner** | Command / Event / 表结构 / 前后端类型命名一致；变更同步 PRD、架构、任务表 |
| **测试** | 用例与验收勾选、回归范围、缺陷分级；不写业务代码 |

## 2. 迭代阶段闸门（Stage Gates）

每个里程碑建议按顺序通过下列闸门；**未通过不得进入下一阶段**（紧急修复除外，须在 issue 或发布说明中备案）。

| 阶段 | 准入条件 | 产出物 |
|------|----------|--------|
| **G0 立项** | 路线图条目明确；本迭代目标一句话可复述 | 迭代目标说明（可写在 issue 或 PRD 修订记录） |
| **G1 需求冻结** | PRD 含：目标用户、必做/不做、P0/P1 分级、与上期已知限制衔接 | 带版本号与日期的 PRD |
| **G2 架构对齐** | 架构文档与 PRD P0 清单一致；禁止项与依赖关系无冲突 | 架构文档版本号更新；必要时「愿景 vs 本迭代落地」对照表 |
| **G3 任务与排期** | 模块依赖 DAG 已定；验收条目可测试；接口 Owner 已指定 | `研发任务与验收计划` 定稿 |
| **G4 开发完成** | P0 功能联调通过；`cargo test` / 约定单测全绿；无未关闭 P0 缺陷 | MR/PR 通过评审；测试包可构建 |
| **G5 测试签收** | 自动化 + 约定手动 E2E 完成；性能抽样或基线已填（若本迭代要求） | 勾选后的验收表导出或截图归档 |
| **G6 发布** | 版本号三处一致；DMG/产物可重现构建；Release Note + 已知限制 | Tag、GitHub Release（若适用）、文档索引更新（按需） |

## 3. 产品设计规范

### 3.1 PRD 必备章节（本迭代范围）

1. **背景与问题**：解决什么、不解决什么（一句话与展开均可）。  
2. **本迭代使命**：与三期路线图中的位置对齐。  
3. **用户与场景**：主路径用户、可选边缘场景。  
4. **功能清单**：按 **P0（必交付）/ P1（可选）/ 明确延后** 分级；**「本迭代不做」** 单独列表。  
5. **边界与约束**：产品层、UI 层、合规/隐私层分别写清。  
6. **验收原则**：与任务表中的 Given/When/Then 同源，避免两套标准。

### 3.2 防范围蔓延

- 新增需求默认进 **下一迭代** 或 **P1 池**；进入本迭代 P0 须产品 Owner 签字（或 issue 批准）并更新 PRD 版本号。  
- PRD 中的长期愿景（如多引擎、多端）须附 **「本迭代落地子集」** 表，避免研发误读全量愿景为当期交付。

### 3.3 版本与变更记录

- PRD 文首维护：**版本号、日期、取代/关联文档**。  
- 重大范围变更：在文首或文末 **修订记录** 中写清日期与摘要。

## 4. 技术架构规范

### 4.1 与 PRD 的关系

- 架构文档是 **实现层唯一基准**；PRD 的 P0 必须在架构中有对应模块、表、接口或明确「本迭代不实现但预留」。  
- 架构 **禁止项**（如一期零外网）若需突破，须走 **G1 变更**：更新 PRD + 架构 + 隐私/合规说明。

### 4.2 模块与风险分级

- **P0**：核心链路；缺失即迭代失败。  
- **P1**：增强能力；须 **可独立关闭** 且 **核心链路不依赖**。  
- **延后**：仅保留接口或 Schema 扩展点，不写业务逻辑（除非单独立项）。

### 4.3 核心工程原则（沿用一期有效实践）

- **统一写入**：数据库写入经单一 Writer（或等价单写者），避免多线程竞争。  
- **宽表 + 聚合**：原始事件与会话/摘要分层，语义在架构中写死。  
- **隐私默认值**：采集字段、脱敏规则、黑名单与 PRD 一致并可测。

### 4.4 接口与数据变更控制

- 任何 **Command / Event / 表结构 / 公共类型** 变更须经 **接口契约 Owner** 同步：  
  - Rust 实现  
  - 前端 `services` / `types`  
  - `研发任务与验收计划` 与测试文档中的条目  
- 建议：变更合并前 checklist 勾选「三处已对齐」。

## 5. 任务排期规范

### 5.1 依赖图优先于周计划

- 先绘制 **模块依赖 DAG**（何模块阻塞何模块），再排周/迭代。  
- 文档中明确 **硬依赖**（不可颠倒）；周次表仅为示例，**以依赖为准**调整。

### 5.2 任务表结构

- 按模块（如 M1、M2…）拆分；每模块包含：**目标、角色分工、Given/When/Then 验收表**。  
- **联调点** 单独标出（例如 Capture 与 Session 归属、权限与 Tracker）。

### 5.3 完成定义（Definition of Done，建议写进迭代约定）

- 代码已合并默认分支；评审通过。  
- 相关 **自动化测试** 已补充或已有测试仍通过（见测试规范）。  
- 接口与类型已与契约 Owner 对齐。  
- 用户可见行为与 **用户使用手册 / 已知限制** 一致或已更新文档。

### 5.4 缓冲

- 含 **系统权限、截图、IPC** 的迭代至少预留 **联调 + 缺陷修复** 缓冲；不在排期上把「示例周次」当作承诺。

## 6. 研发与测试规范

### 6.1 自动化测试边界

- **应自动化**：纯逻辑、迁移幂等、Writer 事务、脱敏、聚合规则、与 PRD 可映射的单元/集成测试。  
- **默认手动**：强依赖 **本机权限、GUI、多应用切换** 的路径；须写入手动 E2E 步骤表（环境、步骤、预期）。

### 6.2 测试文档要求

- 维护 **「测试用例 ↔ PRD/任务条目」** 对照表（参见 `M9_验收与测试.md` 模式）。  
- **发布前**：至少执行 **全量自动化 + 主路径手动 E2E**；改动 DB/Writer/迁移后必跑 `cargo test`（或项目约定命令）。

### 6.3 性能与稳定性

- 对性能有要求的迭代：使用 **统一记录模板**（日期、构建类型、机型、CPU/内存、备注）。  
- 与架构中的目标量级对比；偏离须在备注中说明原因（如 dev 构建、权限弹窗干扰等）。

### 6.4 发布前 Checklist（建议每次发版过一遍）

- [ ] Command/Event 命名与前端封装一致。  
- [ ] 与 PRD 冲突项已落地或已书面备案。  
- [ ] **已知限制**（权限、未签名二进制等）已写入 Release 或用户文档。  
- [ ] P0 验收项已勾选并有归档（截图或导出的表格）。

## 7. 运维与交付规范

### 7.1 版本号

- **唯一来源**：以 `tauri.conf.json`（及项目约定的前端 package 等）为准；脚本化 bump，避免手写多处不一致。

### 7.2 构建与产物

- 发布构建命令、DMG 命名规则以 **`docs/DMG打包指南.md`** 与 `scripts/release.sh` 为准。  
- 新成员环境依赖（如 `jq`、`create-dmg`、`gh`）在打包文档中保持更新。

### 7.3 Release Note

- 每条 Release 建议包含：**新增 / 修复 / 破坏性变更 / 已知限制 / 升级说明**（若有迁移）。  
- 与 PRD「本迭代不做」区分：**已知限制** 是当前版本客观约束，不是未来承诺。

### 7.4 自动更新与遥测

- 默认 **不启用** 自动更新与第三方遥测，除非 PRD 与架构已明确纳入且用户可感知同意；启用前单独立项评审。

---

## 8. 复盘与持续改进

- **每个里程碑结束后** 召开短复盘（可与 G6 后合并）：对照本规范检查哪些闸门执行到位、哪些被跳过及原因。  
- 更新本规范：在文末 **修订记录** 增加版本号、日期与摘要；重大流程变更同步通知全员。
