# TimeLens Windows 兼容 — 技术方案

> **文档性质**：设计与改造说明，供研发与评审使用。  
> **版本**：v1.0 · 2026-03-30

---

## 1. 背景与目标

### 1.1 背景

当前 TimeLens P0 实现以 **macOS** 为主：`acquisition` 仅实现 `macos` 模块；`capture` 依赖 `/usr/sbin/screencapture`；`lib.rs` 中 Tracker/Capture 线程与系统托盘仅在 macOS 启用；部分 API 与存储代码在非 macOS 下无法编译或无法工作。

### 1.2 目标

- **功能**：Windows 10 / 11 上完成 P0 闭环 — Tracker（前台窗口 + AFK）→ Capture（WebP + 去重）→ Aggregation → Writer → IPC → 验证面板。
- **体验**：系统托盘与 macOS 对齐；权限类信息通过现有事件与 Command 下发，**字段名不变**。
- **分发**：绿色 ZIP；**代码层与「安装版」无分叉**，仅打包与文档不同。

### 1.3 非目标（明确不做）

- 浏览器地址栏 URL 采集（UI Automation 等）— **超出 P0**。
- Linux 支持。
- 数据目录与 exe 同级的「便携数据路径」— 除非后续单独立项。
- 修改 `PermissionStatus` 的 JSON 字段名。

---

## 2. 技术原则

1. **厚共享、薄平台**：M1 / M3 / M5 / M6 / M8 业务逻辑与 SQLite 模型保持跨平台；仅在 **OS 集成层**（采集、截图、系统设置跳转、托盘）按 `target_os` 分叉。
2. **统一对外契约**：`FrontWindowState`、`CaptureSignal`、`PermissionStatus` 等类型不变；Windows 对缺省语义用文档说明（如 `bundle_id` 恒为 `None`）。
3. **截图双路径降级**：优先现代 API（如 Windows Graphics Capture），失败降级 GDI（BitBlt），以兼顾 Win10 / Win11。
4. **权限语义映射**：Windows 无「辅助功能 / 屏幕录制」同名项；后端用可探测能力映射到现有两字段，前端用 Windows 专用文案解释。

| 可选清理 | `Cargo.toml` 中 `libc` | 根依赖 | 可移至 macOS-only，减少噪音 |

## 4. 模块级设计

### 4.1 Acquisition（Windows）

- **前台窗口**：`GetForegroundWindow`、`GetWindowTextW`、`GetWindowThreadProcessId`；进程名通过 `OpenProcess` + `GetModuleBaseNameW`，失败则 `QueryFullProcessImageNameW` 取文件名（多级回退，参考业界常见实现）。
- **输出**：填充 `FrontWindowState`：`app_name`（归一化显示名可与 macOS 共用后续扩展表）、`bundle_id: None`、`window_title`、`is_fullscreen`（P0 可先固定 `false` 或简单矩形判断）。
- **空闲**：`GetLastInputInfo` + `GetTickCount` 计算秒数，与现有 AFK 阈值逻辑对接。
- **权限类函数**：
  - `ax_trusted`：映射为 **「能否稳定获取前台窗口信息」**（例如连续探测成功或单次探测成功，具体阈值在实现中固化并写入本文档附录）。
  - `screen_capture_granted` / `screen_capture_refresh_access`：映射为 **「截图路径是否可用」**（如试采一帧或 GDI 全屏试抓，失败则 false）。

### 4.2 Capture（Windows）

- 在 `capture.rs` 内将「获取原始像素」隔离为平台实现；**后续**缩放、WebP、phash、`WriteEvent::Snapshot` **复用**现有逻辑。
- **多显示器**：P0 与产品对齐 —— 优先 **前台窗口所在显示器** 或 **主屏**；与 macOS「按活动窗口选屏」语义尽量一致，文档中写明差异（若有）。

### 4.3 lib.rs 生命周期

- `spawn_tracker_thread` / `spawn_capture_thread`：在 **Windows + macOS** 启用。
- `screen_ok`：Windows 上在窗口聚焦时可调用 `screen_capture_refresh_access` 的 Windows 实现。
- **托盘**：`TrayIconBuilder` 从 `target_os = "macos"` 扩展为 **桌面端 Windows + macOS**（菜单项与事件与现网 macOS 一致）。

### 4.4 API Commands

- `check_permissions`：返回的布尔值符合 Windows 映射语义。
- `open_accessibility_settings` / `open_screen_recording_settings`：macOS 保持现状；Windows 打开 **相关系统设置页** 或 **内置帮助/FAQ 链接**（实现时选一种并写入用户手册）。

### 4.5 前端

- 使用 `@tauri-apps/api` 的 `platform`（或等价方式）区分 **Windows 文案** 与 **macOS 文案**；**不修改** invoke 参数与 `PermissionStatus` 字段名。
- 按钮「打开设置」在 Windows 上调用同一 command，后端路由到 Windows 行为。

### 4.6 Tauri 配置

- `bundle.targets`：包含 Windows 所需 target（绿色版仍可生成 `msi`/`nsis` 以外的目录输出，以官方 CLI 文档为准）；发布说明以 **ZIP 拷贝 `bundle` 目录** 为主。
- **WebView2**：Win10 可能需用户安装 Evergreen Runtime —— 在用户手册与测试计划中体现；可选后续「固定版本运行时打入目录」作为独立优化项。

## 5. 依赖建议（实现阶段最终锁定版本号）

在 `Cargo.toml` 中新增 Windows 条件依赖，候选：

- `winapi`（features 按需：`winuser`、`processthreadsapi`、`psapi`、`wingdi`、`sysinfoapi` 等），和/或
- `windows` / `windows-sys`（若团队倾向官方 crate），
- 截图：`windows-capture` 或自研 WGC 封装（与降级 GDI 组合）。

具体 crate 与 feature 列表以 PR 为准，本方案仅定方向。

---

## 6. 数据与安全

- 数据路径：`%USERPROFILE%\.timelens\...`（与 `dirs::home_dir()` 一致）。
- 绿色版 **不降低** 安全模型：仍仅本地存储；不在方案中引入网络同步。
- 若未来安装到 `Program Files`，写用户目录仍可避免权限问题（与当前设计一致）。

---

## 7. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Win10 无 WebView2 | 手册注明安装；CI/测试机预装 |
| UWP / 商店应用前台 HWND 行为差异 | 多级回退进程名；记录 Known issue |
| 截图被安全软件拦截 | 日志与 `screen_ok` 为 false；文案引导 |
| 高 DPI / 多屏坐标 | 与 macOS 对齐测试用例；必要时阶段内修正 |

## 附录 A：PermissionStatus 字段语义（Windows）

| 字段 | macOS 含义 | Windows 建议映射 |
|------|------------|------------------|
| `accessibilityGranted` | 辅助功能（AX）信任 | 能否可靠获取前台窗口与标题 |
| `screenRecordingGranted` | 屏幕录制授权 | 截图管线是否可用（试采或等价检测） |

前端展示文案与 macOS 区分，避免用户寻找不存在的「屏幕录制」开关（具体文案见研发落地与手册）。
