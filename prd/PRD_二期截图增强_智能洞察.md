# TimeLens 二期产品设计文档 — 截图增强（视觉上下文 + OCR 深度利用）

> **版本**：v2.0  
> **日期**：2026-04-02  
> **定位**：在 **二期基础（事实层 + 会话与截图预览）** 与 **二期 AI 增强（BYOK、`ai_enhanced` 叙事）** 已定义的前提下，立项 **截图数据的结构化利用与 OCR 深度挖掘**：在严格隐私与 opt-in 前提下，把「已采集的屏幕快照」从**存证与回忆**升级为**可分析、可检索、可叙事的上下文信号**——核心手段是 **本地 OCR 提取屏幕文字**，与一期/二期已有六路信号（窗口标题、App 切换、输入指标、剪贴板、通知、会话时长）融合，将分析粒度从「应用级」跃迁至「任务级」。  
> **前置依赖**：`snapshots`（或等价）持续落盘；会话与时间轴可与截图对齐；`PRD_二期AI增强_智能洞察.md` 中的隐私红线、BYOK 原则 **继续适用**；本迭代 **默认不改变** `daily_analysis` 核心指标公式（见 `二期_分析指标字典.md`），新增字段或派生指标须 **另附字典补丁或标注 P1**。  
> **配套文档（待拆）**：建议后续补齐 `二期截图增强_里程碑与验收计划.md`、`二期截图增强_技术方案.md`（含本地 OCR 引擎选型、多信号融合管线、黑名单与脱敏）。

## 〇、背景与问题

### 〇.1 我们已有什么

二期基础已采集并落盘 **六路信号**：

| 信号 | 能回答的问题 | 盲区 |
| --- | --- | --- |
| 窗口标题 / App 名 | 在用哪个软件 | 同 App 下多任务不可区分 |
| App 切换 | 切换频率与路径 | 不知为何切换、切过去干了什么 |
| 输入指标（键鼠） | 是否在活跃操作 | 不知是在写代码还是在刷微博 |
| 剪贴板 | 复制/粘贴动作 | 碎片化，无法拼出上下文 |
| 通知 | 打断来源 | 不知用户是否真的处理了 |
| 会话时长 + 截图 | 时间段 + 可肉眼回忆的画面 | 截图是「冷数据」：不参与聚合、叙事、检索 |

**共同致命短板**：这些信号只知道「在哪」，不知道「在干嘛」——粒度停留在 **应用级**。

### 〇.2 OCR 为什么是质变

屏幕上的文字是唯一能穿透「应用级盲区」的信号。同一个场景：

- 窗口标题："Chrome - Google Docs" → 只知道在 Google Docs
- OCR 提取："Q2 产品路线图 > 第 3 节 竞品分析" → 知道在编辑哪份文档的哪个章节

**从「用了什么工具」到「做了什么任务」，这是粒度的跃迁**。

### 〇.3 行业验证

| 产品/项目 | 做法 | 验证点 |
| --- | --- | --- |
| **Littlebird**（2026 年 $11M 融资） | 用 OCR + NLP 将屏幕转为可搜索文本，存储减少 ~90% | 市场认可「文本优于截图」路线；隐私优先是融资亮点 |
| **Dayflow**（开源 Mac App） | 本地 AI 理解屏幕上下文，区分深/浅工作 | 验证「本地优先 + 视觉上下文」可行且有社区需求 |
| **aw-watcher-enhanced**（ActivityWatch 社区） | 为开源时间追踪加 OCR + LLM 上下文提取 | 验证开源社区对截图 OCR 的真实需求 |
| **ScreenJournal** | AI 驱动的屏幕录制 + 时间追踪 | 验证「AI + 屏幕内容 → 时间分析」有付费用户 |
| **Rize 等隐私优先产品** | 明确拒绝截图，只用 App 级数据 | 反向验证：纯 App 级分析是市场 **下限**，截图增强是 **差异化上限** |

### 〇.4 本迭代要解决的问题（一句话）

在 **用户明确同意** 且 **可撤回、可降级** 的前提下，通过 **本地 OCR** 将截图转化为结构化文本信号，与已有六路数据融合，将分析粒度从「应用级」提升到「任务级」——支撑更深层的复盘叙事、语义检索与自动化工时归属。

## 二、从截图出发：用户使用场景

下列场景按 **发生频率 × 价值** 排序，供优先级裁剪时参考。

### 2.1 基础场景（截图视觉层）

| 编号 | 场景 | 用户行为与心理 | 截图能补上的信息 |
| --- | --- | --- | --- |
| S1 | **日终复盘「这一段到底是啥」** | 看时长条想起模糊，点开会话仍不确定 | 界面元素、文档标题区、任务看板列、会议窗口布局 |
| S2 | **同 App 多任务混淆** | 浏览器/IDE 单窗口标题变化慢 | 可见标签、侧边栏、编辑器路径面包屑、设计画布内容类型 |
| S3 | **深度工作 vs 浅度浏览** | 用户自我认知与统计有落差 | 阅读长文、写代码、刷信息流在视觉密度与布局上可区分（粗粒度即可） |
| S4 | **中断与返工** | 「被打断后回来干了啥」难串联 | 错误弹窗、通知角标、未保存提示、调试面板是否常驻 |
| S5 | **取证式回忆（个人）** | 找「那天截过的那张图/那个界面」 | 按时间、按会话、按 **可选** 文本/语义检索缩略图 |
| S6 | **输出型工作展示** | 写报告、做幻灯片、回邮件 | 可见是否处于「成稿界面」vs「资料搜集」 |
| S7 | **合规与自我保护意识** | 担心敏感屏被上传 | 需要 **可见的黑名单、红区、仅本地** 的默认策略 |

### 2.2 OCR 深度场景（文字提取 + 多信号融合层）

以下场景 **必须** 在 OCR 提取屏幕文字后，与已有信号融合才能实现：

| 编号 | 场景 | 仅靠已有信号的体验 | OCR 融合后的体验 | 融合的信号组合 |
| --- | --- | --- | --- | --- |
| S8 | **自动任务/项目归属** | "VS Code 6h" — 不知在做哪个项目 | "PROJ-1234（用户认证模块）3.5h，PROJ-5678（支付集成）2h，代码评审 0.5h" — 自动按 Jira 票号 / Git 分支 / 文档标题归属 | OCR（票号/分支名/文档标题） + 窗口标题 + 会话时长 |
| S9 | **工作流链路重建** | "Chrome → VS Code → Chrome → Terminal" — 切换记录是碎片 | "阅读 Jira 需求 → IDE 编写 auth 模块 → 运行测试（passed）→ 回 Jira 更新状态" — 自动拼出完整工作流叙事 | OCR（页面内容/终端输出） + App 切换序列 + 时间戳 |
| S10 | **深度/浅度精准判定** | "Chrome 活跃 60min" — 粗粒度 | "阅读 Rust 文档 45min（深度学习）+ 刷 Twitter 15min（浅度浏览）" — 同一个 App 内按内容拆分深浅 | OCR（页面文字密度/内容类型） + 输入指标（键盘节奏） + 会话时长 |
| S11 | **阻塞与卡点检测** | 输入指标显示「鼠标频繁移动但少量输入」— 无法确认原因 | "在 webpack 编译错误上卡了 40min：反复搜索 'Module not found'" — 识别到错误弹窗/终端报错/反复搜索同类关键词 | OCR（错误信息/搜索框内容） + 输入指标（无效操作模式） + 会话时长 |
| S12 | **全文语义检索** | 只能按 App 名或日期翻找 | "那篇关于 Rust 生命周期的文章" → 直接命中、返回截图与时间点 | OCR 全文索引 + 时间戳 |
| S13 | **会议内容关联** | "Zoom 打开了 2.5h" — 不知几场会、聊了什么 | "产品评审 1h（共享了设计稿）+ 团队站会 30min + 1:1 1h" — 从会议界面提取标题、参会人数、是否共享屏幕 | OCR（会议标题/参会列表） + 通知数据 + 窗口标题 |
| S14 | **自动周报/工时单草稿** | 用户手动回忆整周做了什么 | 系统自动汇总："本周在 3 个项目上投入时间——认证模块 12h、支付 8h、文档 5h；会议 6h；代码评审 3h" — 直接可用于填工时或写周报 | S8 任务归属聚合 + S13 会议关联 + 日/周维度汇总 |
| S15 | **学习 vs 产出 vs 沟通 细分** | Intent 只能粗分 coding/meeting/browsing | "编写代码 4h / 阅读文档（学习）2h / 代码评审（协作）1h / 邮件沟通 1h" — 同属 coding 的「写」和「读」、同属 browsing 的「学习」和「摸鱼」被区分 | OCR（内容类型识别） + Intent + 输入指标 |

## 三、需求分层：显性、隐性、真实

### 3.1 显性需求（用户会直接说出来）

| ID | 显性诉求 | 产品回应方向 |
| --- | --- | --- |
| E1 | 「日报别只写 App 名，说说我在干啥」 | OCR 提取屏幕文字 → 自动任务归属 → `ai_enhanced` 可附带脱敏后的 **任务级摘要**（不编造事实数字） |
| E2 | 「我能搜到某天某个界面吗」 | OCR 文本本地全文索引 → 按关键词/语义搜索 → 命中帧 + 缩略图 + 时间点 |
| E3 | 「千万别把我密码、聊天、银行界面发出去」 | 应用黑名单、OCR 敏感词检测（密码框/银行域名/身份证号模式）、默认不出境 |
| E4 | 「别占太多盘、别太卡」 | OCR 只存结构化文本（比截图小 ~90%）、抽样策略、异步处理、可清理 |
| E5 | 「能不能自动帮我填工时单」 | OCR 识别 Jira/Linear/Notion 票号 → 按项目自动聚合时间 → 导出周报/工时草稿 |
| E6 | 「我想知道我卡在哪个问题上花了多久」 | OCR 检测错误弹窗/编译失败/重复搜索 → 阻塞时长统计 → 日报中标注 |

### 3.2 隐性需求（用户很少明说，但会表现为流失或吐槽）

| ID | 隐性信号 | 产品回应方向 |
| --- | --- | --- |
| I1 | 不信任「自动分类」 | 所有 OCR 推断 **带置信度/可关闭**；与 Intent 纠错联动；用户修正后学习 |
| I2 | 觉得「被监视」 | **默认本地**、**逐能力授权**、文案避免「监控」，强调「自助复盘」；OCR 原文 **不展示完整句子**，只展示关键词/标签 |
| I3 | 不想额外操作 | **复用已有截图**，OCR 后台异步处理，零用户操作即可获得任务级报告 |
| I4 | 怕以后无法反悔 | 一键清除 OCR 索引、按日删除、与总开关联动 |
| I5 | 分类不准时反而更烦 | 宁可显示 unknown 也不乱标；OCR 置信度低时 **静默降级** 为 App 级粒度 |
| I6 | 「说好只做个人工具，别变成老板监控」 | 无远程查看、无团队管理端、数据不出本机除非 BYOK 主动授权 |

### 3.3 真实需求（Jobs-to-be-done）

> **JTBD-1**：**当** 我回顾一段时间的使用记录时，**我想** 在几分钟内重建「当时的情境与任务性质」，**以便** 做计划调整、写周报、或说服自己「今天并非虚度」——**而不必** 逐条手工标注或依赖不可靠记忆。

> **JTBD-2**：**当** 我需要向自己或团队交代时间去向时（填工时、写周报、回溯某个问题），**我想** 系统直接给我一份「按任务/项目/类型」的时间分布草稿，**以便** 我只需微调而非从零回忆。

> **JTBD-3**：**当** 我怀疑自己在某个问题上浪费了太多时间时，**我想** 知道具体卡在了什么上、卡了多久，**以便** 下次更早寻求帮助或换思路。

截图增强的本质交付物：**从「在哪个 App」到「做了什么任务」的粒度跃迁**，让复盘从「看着数字猜」变成「看着事实懂」。

## 四、功能与能力清单

### 4.1 原则

| 原则 | 说明 |
| --- | --- |
| **增强不替代** | OCR 与视觉信号 **不覆盖** `daily_analysis` 既有公式；新增「任务级」维度作为 **可选增强层**，降级时回退到 App 级 |
| **文本优于图片** | 优先传递 OCR **结构化文本摘要**（关键词、标签、票号），而非原始截图；存储小、隐私风险低、可索引（参考 Littlebird 路线） |
| **默认本地** | OCR 处理、索引构建、任务归属推断 **全部在本地完成**；出境严格 opt-in |
| **可撤回** | 关闭开关后停止新处理；可一键删除已生成的 OCR 索引与任务归属数据 |
| **黑名单优先** | 与二期 AI 增强 PRD 黑名单一致；OCR 增加 **敏感词模式检测**（密码框、银行域名、证件号正则）作为第二道防线 |
| **宁缺勿乱** | OCR 置信度低时标记 unknown，**不强行归类**；不在 UI 中以确定语气呈现不确定推断 |

### 4.2 P0 — 基础层：OCR 管线与安全（本迭代必达）

| ID | 能力 | 说明 | 对应场景 |
| --- | --- | --- | --- |
| P0-1 | **截图安全策略产品化** | 应用/域名/窗口类黑名单 + **OCR 敏感词二次过滤**（命中密码框特征、银行域名、身份证号模式的帧 → 丢弃 OCR 结果、标记为 `redacted`）；隐私说明更新 | S7 |
| P0-2 | **本地 OCR 引擎** | 对未命中黑名单的帧执行 **本地 OCR**（技术方案选型另册：macOS Vision / Tesseract / 等价方案）；提取结构化文本块 **仅存本地 SQLite**；原始截图可按策略清理，OCR 文本保留（体积约为截图的 1/50~1/100） | S8-S15 |
| P0-3 | **会话级 OCR 摘要数据结构** | `session_ocr_context` 表/字段：提取的关键词片段、识别到的实体（票号/文件路径/文档标题/URL 域名/会议标题）、场景粗标签（coding / reading / communication / meeting / debugging / media / unknown）、置信度 | S1-S6 |
| P0-4 | **复盘 UI：OCR 增强摘要** | 会话详情中展示 **一行任务级摘要**（如「编辑 auth 模块 · VS Code」而非仅「VS Code」）；可关闭；与 `fact_only` 报告不冲突 | S1, S2 |
| P0-5 | **与 AI 增强的衔接** | BYOK + 附图开关均开时，向模型发送 **OCR 结构化摘要文本**（而非原始图像）——体积小、隐私可控、效果足够；产品要求 **最小必要** | — |

| P1-5 | **全文语义检索** | OCR 文本本地全文索引 → 支持按关键词/自然语言搜索 → 返回匹配帧的缩略图 + 时间点 + 会话上下文 | S12 | OCR 索引 + 时间戳 |
| P1-6 | **会议智能关联** | OCR 从会议软件界面提取会议标题、参会人数、是否共享屏幕 → 结合通知数据 → 自动拆分「今日 3 场会议，共 2.5h」 | S13 | OCR（会议界面） + 通知 + 窗口标题 |
| P1-7 | **自动周报/工时单草稿** | 聚合 P1-1 任务归属 + P1-6 会议关联 → 按周维度汇总 → 导出 Markdown 草稿（用户微调即可提交） | S14 | P1-1 + P1-6 聚合 |

### 4.5 本迭代明确不做

| 不做 | 原因 |
| --- | --- |
| 实时屏幕直播、远程查看 | 非个人复盘范畴，合规与信任成本过高 |
| 自动识别密码、证件、具体聊天对象 | 隐私与法律风险；应进入黑名单或遮罩 |
| 替代用户做绩效评判、向第三方证明工时 | 产品定位是自助工具，非监工系统 |
| 无 BYOK 的内置云端多模态 Key | 与二期 AI 增强原则冲突 |
| 完整 NLP 语义理解（意图推理、情感分析等） | 超出 OCR 文本提取的范畴，可在三期借助 LLM 增强 |

## 五、边界与约束

### 5.1 隐私与出境

| 层级 | 规则 |
| --- | --- |
| **默认行为** | OCR 处理、索引构建、任务归属推断 **全部在本地完成**；任何 OCR 文本与摘要 **仅存本机 SQLite**，不上云 |
| **黑名单第一道** | 命中黑名单的帧 → **跳过 OCR**，标记 `ocr_status = redacted`；黑名单与二期 AI 增强 PRD 一致 |
| **敏感词第二道** | OCR 完成后对文本块执行正则扫描（密码框模式、银行/支付域名、身份证号、手机号）→ 命中则丢弃该文本块，整帧降级为 `redacted` |
| **出境（opt-in）** | 须用户显式勾选「允许 OCR 摘要辅助 AI 叙事」；传输内容为 **关键词/标签/票号**（结构化摘要），**禁止**传输 OCR 原始全文或截图 Base64；日志不得包含任何用户屏幕内容 |
| **与 AI 增强对齐** | 允许出境的仍是「最小必要」结构化摘要；`raw_events` 全表、完整窗口标题原文 **严格禁止出境** |

### 5.2 性能与存储

| 约束 | 目标值（建议，技术方案可调整） |
| --- | --- |
| OCR 处理时机 | **异步后台**，不阻塞采集主路径 |
| 采样间隔 | 可配置（建议默认 30s/帧，高密工作模式可降至 10s） |
| 单帧 OCR 耗时 | < 500ms（本地 Vision API / Tesseract；具体基准由技术方案测定） |
| OCR 文本存储体积 | 约为对应截图的 **1/50 ~ 1/100**（无需保留原图即可索引） |
| 数据保留策略 | OCR 文本/摘要可配置保留天数（独立于截图保留策略）；支持一键清除 |

### 5.3 正确性与降级

- OCR 置信度低（< 阈值）时，场景标签标记为 `unknown`，**不强行归类**；UI 不以确定语气呈现不确定推断。
- 任务归属推断失败 → **静默降级**为 App 级粒度；报告中不产生空白或错误条目。
- OCR 模块崩溃或超时 → **不影响** `fact_only` 报告与 `daily_analysis` 生成（硬隔离）。
- `ai_enhanced` 中引用 OCR 推断时，建议标明「基于屏幕内容推测」，与事实数字 **分块排版**。

## 六、竞争力分析：OCR 多信号融合带来的差异化

| 维度 | 纯 App 级产品（RescueTime / Rize 等） | **TimeLens 截图增强后** |
| --- | --- | --- |
| 分析粒度 | 应用级（"VS Code 6h"） | **任务级**（"认证模块 3.5h + 支付集成 2h"） |
| 工时归属 | 手动填写 | **自动按票号/项目名聚合**，导出草稿 |
| 深度工作判定 | 单 App 停留时长 > 阈值 | **同 App 内区分**：写代码 vs 刷文档 vs 看视频 |
| 阻塞检测 | 不支持 | **识别卡点**：错误弹窗 + 重复搜索 + 低有效输入 |
| 回忆检索 | 按时间/App 名筛选 | **按屏幕文字全文搜索**，命中截图与时间点 |
| 隐私策略 | 全量截图上云（多数产品）或仅 App 名本地 | **OCR 文本本地**，出境仅结构化摘要，存储小 ~90% |
| 周报自动化 | 不支持 | **自动汇总任务 + 会议 → Markdown 草稿** |

**核心差异化定位**：在「完全隐私友好的本地工具」与「功能完整的 AI 助理」之间，TimeLens 走的是 **本地 OCR + 多信号融合** 的中间路线——隐私成本低、分析深度高、无需外网也能完成任务级洞察。

## 七、验收原则（摘要，与里程碑表同源扩展）

详细 Given/When/Then 建议在 **`二期截图增强_里程碑与验收计划.md`** 中展开；本 PRD 只列主轴。

| 主题 | 验收要点 |
| --- | --- |
| **默认安全** | 新装或升级后，OCR 开关、视觉索引、出境附图均为**关**；开启前有独立说明与确认弹窗 |
| **黑名单生效** | 配置黑名单后，对应帧 `ocr_status = redacted`，**不参与**索引构建与出境；既有索引不受影响（历史数据清理由产品定） |
| **敏感词过滤** | 含密码框/银行域名/证件号模式的帧，OCR 结果丢弃，UI 不展示任何文字摘要 |
| **本地摘要** | 开启本地 OCR 且有有效数据时，会话详情展示**任务级一行摘要**；关闭后立即停止更新（已生成摘要建议标注「已关闭」） |
| **任务归属** | P1-1 开启时，含已知票号模式的会话，日报中出现按项目聚合的时间分布；无识别结果时不产生空条目 |
| **降级** | OCR 模块崩溃或超时，`fact_only` 与 `daily_analysis` 正常生成，报告无 OCR 相关段落 |
| **AI 衔接** | 仅当 BYOK + OCR 出境开关均开时，外向请求才携带 OCR 结构化摘要；否则与二期 AI 增强行为完全一致 |
| **存储与清理** | 一键清除 OCR 索引后，检索与摘要功能回退到无 OCR 状态；清除操作不影响截图与 `daily_analysis` |
