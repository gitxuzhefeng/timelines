# PRD：十期 — AI 对话助手（V10）

**版本**：v1.0 · 2026-04-21
**上游基准**：`prd/PRD_九期_数据导出与本地报告.md`
**关联实现**：桌面端 Tauri + React（`project/`）

---

## 1. 背景与问题

一至九期建立了完整的本地数据资产：窗口会话、OCR 全文、每日/每周分析报告、键盘指标、专注评分。但所有这些数据都以"被动展示"为主——用户需要主动翻看页面、对照图表才能得出结论。

用户真实遇到的问题是：

- 「我这周到底在哪个项目上花了多少时间？」——需要反复翻日报手动累加
- 「上个月哪几天是我状态最好的？」——周报没有月视角，没有快捷路径
- 「我最近一直在做的那个任务叫什么来着，上周四我在哪个软件里工作最久？」——OCR 搜索可以找，但不够自然
- 「我今天的状态怎么样，接下来要注意什么？」——数据有，但没有人帮你解读

竞品 Work_Review 的工作助手功能解决了"数据可问答"的核心需求，是其最直观的 AI 价值感知入口。但竞品的助手基于通用问答模式，回答质量依赖用户问题的精确度，且无法感知 TimeLens 独有的深度指标（专注评分、心流指数、碎片化率）。

**本期要做的不是"另一个通用 AI 聊天框"，而是一个懂你工作数据的专属分析师。**

## 3. 用户与场景

### 3.1 核心用户画像

- **复盘型用户**：每周五或每月初想快速回顾，不想逐页翻数据
- **目标追踪型用户**：设定了"每天深度工作 4 小时"的目标，想随时知道自己距离目标多远
- **效率优化型用户**：想找规律——我状态最好的时间段在哪里，什么应用最消耗我的时间

### 3.2 主路径场景

**场景 A（最常见）**：下午结束工作，用户点开助手说「今天的状态怎么样」，助手回答今日专注时长、心流评分，并指出「下午 3 点后碎片化明显上升，建议明天保持上午的节奏」，同时附一个「查看时间线」的跳转按钮。

**场景 B（跨日回顾）**：用户问「上周我在 Cursor 上花了多少时间」，助手从 `weekly_analysis` + `window_sessions` 中精确计算，回答「上周共 18.4 小时，主要集中在周二和周四上午」，并给出与前一周的对比。

**场景 C（OCR 语义查找）**：用户问「上周我看过一篇关于 Rust 异步的文章，在哪里」，助手通过 OCR 全文搜索定位截图，回答「周三下午 2:34 在 Firefox 中，标题含 async/await」，附跳转到时间线该时刻的按钮。

**场景 D（主动洞察）**：用户打开助手页，助手不等用户提问，主动推送一条「今日值得注意」——「今天你的心流评分是近 14 天最高，比上周同期高 23%，当前状态很好，建议利用下午前 2 小时做最重要的工作」。

### 3.3 边界

- 本期不做语音输入。
- 不支持连续多轮深度对话（单次问答为主，保留最近 10 条历史供参考，不做长期对话记忆）。
- 助手只访问本地数据，不引入外部搜索或联网能力。
- 用户隐私保护：传给 AI 的上下文严格遵循二期隐私红线——只传 `daily_analysis`/`weekly_analysis` 聚合 JSON、应用名、intent 名；不传窗口标题原文、OCR 内容原文（OCR 场景单独处理，见 §4.3）。
- 不做团队/多用户维度。

## 4. 功能与能力清单

### 4.1 P0（本迭代必达）

| 能力 | 说明 |
|------|------|
| 助手导航入口 | 左侧导航新增「助手」菜单项（位于「周报」与「设置」之间），路由 `/assistant` |
| 对话输入框 | 页面底部固定输入框，支持 Enter 发送、Shift+Enter 换行、发送中禁用输入、AI 流式输出（stream） |
| 对话历史展示 | 会话列表展示最近 10 条问答，区分用户/助手气泡，支持 Markdown 渲染 |
| 主动洞察卡片 | 每次进入助手页，若当日 `daily_analysis` 已生成，页面顶部展示一张「今日洞察」卡片（由本地规则引擎生成，不调 AI，0 token 消耗），内容包含：今日专注时长 vs 近 7 天均值、当前心流评分段位、最重要的 1 条建议 |
| 自然语言时间解析 | 支持「今天/昨天/本周/上周/本月/上个月/最近 N 天」等时间表达，映射到具体日期范围后查询 |
| 结构化数据查询 | 可查询 `daily_analysis`（专注时长、心流评分、碎片化率、深度工作时长）和 `weekly_analysis`（周度聚合指标）；支持跨日聚合（如"上周一共"） |
| 应用使用查询 | 可查询指定时间内某应用/某类 intent 的使用时长，数据来源 `window_sessions` |
| 跳转联动 | 助手回答中可附带「跳转」操作按钮（查看时间线、查看日报、查看周报），点击后跳转到对应页面并自动定位到相关日期 |
| AI 不可用降级 | 若 AI 渠道未配置或请求失败，主动洞察卡片仍正常显示（本地规则生成）；对话框展示「AI 服务暂不可用，请在设置中检查 AI 配置」，不白屏 |

### 4.2 P1（可并入本期）

| 能力 | 说明 |
|------|------|
| OCR 语义查找 | 用户询问"我看过 XX"类问题时，助手通过 `search_ocr_text` 接口检索，将结果脱敏后（仅返回时间戳、应用名、截图缩略描述）作为上下文传给 AI 回答；原始 OCR 文本不直接出境 |
| 对话历史持久化 | 将对话历史存入本地 `assistant_history` 表，支持跨会话查看最近 30 条 |
| 快捷问题入口 | 主动洞察卡片下方展示 3 个快捷问题按钮（「今天专注了多久」「本周最高效的一天」「最近状态趋势」），点击直接发送 |

### 4.3 明确不做（本迭代）

| 项目 | 说明 |
|------|------|
| 连续长对话记忆 | 超过 10 条历史不进入上下文窗口，不做对话摘要压缩 |
| OCR 原文出境 | OCR 内容只在本地检索定位，脱敏后仅传时间/应用等元数据给 AI |
| 语音输入 | 延后 |
| 自定义助手角色/提示词 | 延后 |
| 推送通知提醒 | 不做「每日主动洞察」系统通知推送，用户需主动打开助手页 |

## 5. 差异化设计：我们 vs 竞品

| 维度 | 竞品 Work_Review | TimeLens 十期 |
|------|-----------------|--------------|
| 上下文深度 | 通用问答，基于原始记录拼接上下文 | 专属指标体系：心流评分、碎片化率、深度工作时段等独有维度作为上下文 |
| 主动性 | 被动等待用户提问 | **主动洞察卡片**（本地规则引擎，0 token），每次打开有"第一句话"，不需要用户想问题 |
| OCR 联动 | OCR 搜索独立，与问答分离 | 「我看过 XX」直接触发 OCR 语义查找，问答和数据检索融合 |
| 跳转联动 | 回答是纯文本 | 回答中附带跳转按钮，查询结论即操作入口，闭环更短 |
| 隐私 | 依赖用户信任，细节不明 | 严格遵循隐私出境规则，传 AI 的内容经过聚合脱敏，用户可在设置中查看"本次传给 AI 的内容" |

## 6. 数据定义

### 6.1 对话历史表（`assistant_history`）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK | 自增 |
| `role` | TEXT | `"user"` / `"assistant"` |
| `content` | TEXT | 消息正文（Markdown） |
| `context_snapshot` | TEXT | 本次问答传给 AI 的上下文摘要（JSON，用于隐私透明，只记录字段名和范围，不记录原始值） |
| `created_at` | TEXT | 消息时间戳 |

### 6.2 主动洞察规则（本地引擎，无 AI 调用）

主动洞察卡片由前端本地规则引擎生成，规则优先级从高到低：

| 优先级 | 触发条件 | 洞察文案模板 |
|--------|---------|-------------|
| 1 | 今日 `flow_score` > 近 14 天均值 × 1.2 | 「今天心流评分 {score}，是近两周最好状态之一，适合处理重要任务」 |
| 2 | 今日 `fragmentation_pct` > 60% | 「今天碎片化率 {pct}%，注意力分散，建议关闭通知专注 30 分钟」 |
| 3 | 今日 `deep_work_minutes` < 60 且当前时间 < 18:00 | 「今天深度工作时长不足 1 小时，下午还有时间，试试关闭聊天软件专注一段」 |
| 4 | 今日 `total_active_ms` > 近 7 天均值 × 1.3 | 「今天工作时长 {h}h，超过近一周均值 30%，注意休息节奏」 |
| 5 | 默认（以上均不触发） | 「今日专注时长 {h}h，心流评分 {score}，继续保持」 |

### 6.3 AI 上下文构建策略

每次对话调用 AI 时，上下文按以下层级拼装，总 token 预算控制在 2000 以内：

| 层级 | 内容 | Token 估算 |
|------|------|-----------|
| 系统提示 | 角色定义 + 隐私约束（中/英文，跟随界面语言） | ~200 |
| 用户档案摘要 | 近 7 天 `daily_analysis` 聚合：均值、峰值、低谷 | ~300 |
| 问题相关快照 | 根据时间解析结果，拉取对应日期的 `daily_analysis` / `weekly_analysis` | ~800 |
| 对话历史 | 最近 3 条历史问答（role + content 截断至 200 字） | ~600 |
| 用户当前问题 | 原始输入 | ~100 |

## 7. 方案摘要（实现约束）

### 7.1 后端（Rust）

1. **数据库迁移**：新增 `assistant_history` 表。
2. **`query_assistant(question: String, context: AssistantContextDto, lang: String)` 命令**：
   - 接收前端构建好的上下文 DTO，向已配置 AI 渠道发起流式 chat completion 请求。
   - 返回 `Stream<Result<String, String>>`，前端消费流式输出逐字显示。
   - 失败时返回明确错误类型（`ai_not_configured` / `ai_request_failed` / `timeout`）。
3. **`get_assistant_context(date_range: (String, String))` 命令**：根据日期范围从 `daily_analysis`、`weekly_analysis`、`window_sessions` 查询并组装 `AssistantContextDto`，不含窗口标题原文和 OCR 内容。
4. **`save_assistant_message(role: String, content: String, context_snapshot: String)` 命令**：写入 `assistant_history`。
5. **`get_assistant_history(limit: u32)` 命令**：返回最近 N 条记录。
6. **时间解析**：时间表达（「今天」「上周」等）在前端解析为具体日期范围，后端只接受 ISO date 格式，不做 NLP。
7. **流式输出**：复用七期 `ai_client.rs` 的 HTTP 客户端，新增 SSE/chunked 流式读取能力；Tauri event channel 将 chunk 实时推送前端。

### 7.2 前端（React）

1. **新增 `AssistantPage`**：路由 `/assistant`，左侧导航新增入口。
2. **`InsightCard` 组件**：页面顶部固定卡片，本地规则引擎（纯 TS 计算，不调后端），读取 `appStore` 中的 `activity` 数据计算洞察。
3. **`MessageList` 组件**：展示对话历史，用户气泡右对齐，助手气泡左对齐，内容 Markdown 渲染，助手消息支持流式追加。
4. **`ChatInput` 组件**：底部输入框，Enter 发送，内置时间表达解析器（正则映射，不依赖 AI）将「昨天」等词解析为日期范围后附加到上下文构建请求。
5. **`ActionButton` 组件**：助手回答末尾可附带操作按钮（「查看时间线」「查看日报」），按钮数据由 AI 回答中的 `[ACTION:timeline:2026-04-20]` 格式标记解析生成，前端拦截渲染为按钮而非纯文本。
6. **流式渲染**：监听 Tauri event channel，每收到 chunk 追加到当前助手消息尾部，流结束后标记消息完成并存库。
7. **i18n**：新增十期相关词条到 `zh-CN.json` 和 `en.json`，主动洞察文案模板双语。

### 7.3 隐私透明设计

助手设置子页（或消息详情展开）中，用户可查看「本次传给 AI 的内容」，展示 `context_snapshot`：只显示字段名、日期范围、条数，不展示原始数值明细。这让用户对"哪些数据出境了"有直接感知。

## 8. 关键指标与验收阈值

### 8.1 验收口径（P0）

1. **导航**：左侧导航出现「助手」菜单项，点击后路由跳转到助手页。
2. **主动洞察卡片**：进入助手页，若当日 `daily_analysis` 已生成，顶部展示洞察卡片；卡片内容随当日数据变化（不同 flow_score 区间显示不同文案）。
3. **对话发送**：输入问题后按 Enter，消息出现在对话列表，助手开始流式响应。
4. **流式输出**：助手回答逐字出现，不是等待后整段刷新。
5. **时间解析**：输入「今天」「上周」等词，助手回答中的时间范围与对应日期吻合。
6. **跳转按钮**：含 `[ACTION:...]` 标记的回答渲染为跳转按钮，点击后正确跳转并定位日期。
7. **AI 不可用降级**：未配置 AI 时，对话框展示明确提示，洞察卡片仍正常显示。
8. **隐私约束**：通过网络抓包或日志确认，发送给 AI 的请求体中不含窗口标题原文、OCR 内容原文。
9. **无回归**：日报、周报、设置、时间线等已有页面功能正常。

### 8.2 性能要求

| 指标 | 目标 |
|------|------|
| 洞察卡片渲染时间 | < 100ms（本地计算，无网络） |
| 上下文构建（`get_assistant_context`） | < 300ms |
| 首个 AI 流式 chunk 到达时间 | < 3s（依赖 AI 渠道，超时阈值 30s） |
| 对话历史加载 | < 200ms |

## 9. 验收要点（Given / When / Then 摘要）

1. **Given** 当日已生成 `daily_analysis`，**When** 用户进入助手页，**Then** 顶部洞察卡片展示当日专注时长和心流评分，文案符合 §6.2 规则逻辑。
2. **Given** 用户输入「今天专注了多久」，**When** 发送后 AI 响应，**Then** 回答中的时长数值与「今日透视」页展示一致。
3. **Given** 用户输入「上周哪天状态最好」，**When** AI 响应，**Then** 回答引用上周 7 天的 flow_score 数据并指出最高分日期。
4. **Given** 助手回答含跳转按钮，**When** 用户点击「查看时间线」，**Then** 跳转到时间线页并定位到对应日期。
5. **Given** AI 渠道未配置，**When** 用户发送消息，**Then** 对话框显示「AI 服务暂不可用，请在设置中检查 AI 配置」，洞察卡片仍正常显示。
6. **Given** 当日 `fragmentation_pct > 60%`，**When** 用户进入助手页，**Then** 洞察卡片显示碎片化告警文案（优先级 2）。
7. **Given** 用户询问「上周我在哪里看过 Rust 相关内容」（P1 OCR），**When** AI 响应，**Then** 助手给出包含时间戳和应用名的定位结果，不展示原始 OCR 文本。
8. **Given** 通过抓包检查 AI 请求体，**Then** 请求中不含任何窗口标题原文字段或 OCR 文本字段。

## 10. 已知限制

- AI 回答质量依赖用户配置的模型能力；建议推荐 DeepSeek / Qwen 等中文能力强的模型用于中文问答。
- 主动洞察卡片基于本地规则，规律较简单，不能替代完整 AI 分析；后续可在十二期引入更复杂的本地推理。
- 跳转按钮依赖 AI 输出中的 `[ACTION:...]` 格式标记；若 AI 不遵循格式，回答为纯文本不含按钮，不影响功能正确性，只影响体验。
- 时间表达解析为正则映射，极端表达（如「前天的前天」）暂不支持。
