## 1. 背景与问题（PRD_BACKGROUND）

十三期为时间线引入了「概括视图」，将全天会话按 30 分钟窗口智能合并，展示每个时段的主要应用和占比。用户反馈概括效果已满足日常回顾需求，但在需要进一步了解某个时段内具体发生了什么时，只能切换到「完整视图」在全天数百条记录中手动定位，缺少从概括到明细的**原地钻取**能力。

同时，当前时间线顶部的 Bridge 卡片（同一口径 / 总时长 / Top 应用）与概括视图信息重叠，用户感知价值较低；活动密度条过细、偏灰，无法一眼看出全天分布。概括卡片和展开后的会话卡片也偏日志式排布，尚未充分突出用户真正关心的内容：**这段时间在干什么、持续了多久、是否连续专注、值不值得进一步看**。

此外，当前仅有「科技风」（深色）和「白色风」（浅色）两种主题，用户希望有更多视觉风格可选，包括流行的编辑器/品牌主题。

## 3. 用户与场景（PRD_USERS）

### 3.1 主路径场景

| 场景 | 用户行为 | 期望体验 |
|------|----------|----------|
| 时段钻取 | 在概括视图中看到「8:00-9:30 VS Code（78%）」，想知道这段时间具体做了什么 | 点击该卡片，卡片下方原地展开 8:00-9:30 内的所有会话明细 |
| 快速收起 | 看完明细后不需要了 | 再点一下卡片，明细收起，恢复概括状态 |
| 多段对比 | 想同时看上午和下午两个时段的明细 | 两张卡片可以同时展开，互不影响 |
| 会话详情 | 在展开的明细中看到某条会话，想看截图 | 点击该会话条目，打开现有的会话详情侧板 |

### 3.2 边界（PRD_SCOPE）

**不做 / 延后**

- 不改变概括视图的合并算法和展示逻辑
- 不改变完整视图的行为
- 展开状态不持久化（页面刷新或切换视图后重置）
- 不做展开区域内的二次筛选或搜索

#### 4.1.8 分组颜色体系：统一由分组配置驱动

**问题**：当前所有与 intent 相关的颜色（活动密度条色块、概括卡片 intent 标签、展开会话 intent 标签）都通过 `intentVisual()` 正则匹配到 4 个固定 bucket（a/b/c/d），再映射到 4 个 CSS 变量。实际使用中，大量分组名（如"编码开发""文件管理""研究检索"）未被正则命中，全部 fallback 到同一个 bucket，导致颜色完全相同，无法区分。

**方案**：废弃正则 bucket 机制，改为**所有 intent 颜色统一从分组颜色配置中读取**。

| 能力 | 说明 |
|------|------|
| 内置分组默认颜色 | 为内置预设分组分配默认颜色，用户未自定义时使用该默认值 |
| 内置分组颜色可修改 | 在应用分组管理页面，内置分组也支持修改颜色（与自定义分组一致） |
| 自定义分组颜色 | 创建自定义分组时选择颜色（已有），颜色在所有 intent 相关 UI 中生效 |
| 颜色查找链路 | intent 名 → 查分组颜色表（自定义分组 `custom_intents.color` + 内置分组默认色）→ 直接用该颜色渲染 |
| 未匹配 fallback | 未找到对应分组的 intent 使用灰色 `#6b7280` |
| 影响范围 | 活动密度条色块、概括卡片 intent 标签、展开会话 intent 标签、完整视图 intent 标签、时间轴轴点颜色 — 所有与 intent 相关的颜色渲染 |

**内置分组默认配色**

| 分组名 | 默认色 | 色值 | 说明 |
|--------|--------|------|------|
| 编码开发 | 青绿 | `#00f5d4` | 深度工作 / 开发 |
| 研究检索 | 紫色 | `#bd93f9` | 浏览 / 学习 / 检索 |
| 通讯沟通 | 蓝色 | `#61afef` | IM / 邮件 / 会议 |
| 其他 | 灰色 | `#6b7280` | 未分类 / 默认 |

**数据流变更**

```
旧链路：intent 名 → intentVisual() 正则 → 4 bucket → CSS 变量颜色
新链路：intent 名 → intentColorMap（内置默认 + custom_intents 表）→ 直接 HEX 颜色
```

**应用分组管理页面变更**

| 能力 | 说明 |
|------|------|
| 内置分组颜色编辑 | 在分组管理面板中，内置分组也显示色点和颜色选择器，用户可修改 |
| 内置分组颜色持久化 | 修改后的内置分组颜色存入 `custom_intents` 表（name 与内置分组名相同，仅存颜色覆盖） |
| 颜色预览 | 分组列表中每个分组名前显示对应颜色色点 |

## 5. 数据依赖（PRD_DATA）

### 5.1 概括卡片与原始会话的映射

当前 `summaryBlocks` 合并算法基于 30 分钟滑动窗口，合并后的 `SummaryBlock` 记录了 `startMs` 和 `endMs`。展开时需要从原始 `sessions` 中筛选出落在该时段内的所有会话：

```
展开会话 = sessions.filter(s => s.startMs < block.endMs && s.endMs > block.startMs)
                   .sort((a, b) => a.startMs - b.startMs)
```

**无需新增后端 API 或数据表**，所有数据已在前端 `sessions` 状态中。

## 6. 前端实现要点（PRD_COMPONENTS）

### 6.1 状态管理

```typescript
// TimelinePage 内部状态
const [expandedBlocks, setExpandedBlocks] = useState<Set<string>>(new Set());

function toggleBlock(blockId: string) {
  setExpandedBlocks(prev => {
    const next = new Set(prev);
    if (next.has(blockId)) next.delete(blockId);
    else next.add(blockId);
    return next;
  });
}
```

### 6.2 会话筛选

```typescript
// 为每个 SummaryBlock 计算其包含的原始会话
function getBlockSessions(block: SummaryBlock): WindowSession[] {
  return sessions
    .filter(s => s.startMs < block.endMs && s.endMs > block.startMs)
    .sort((a, b) => a.startMs - b.startMs);
}
```

### 6.4 视觉与信息架构原则

- **概括卡片先回答“这段时间在干什么”**：时间范围、主应用、时长必须最醒目
- **展开会话先回答“具体发生了什么”**：窗口标题优先于技术字段
- **完整视图保留日志属性**，但不作为概括视图的默认信息模型
- **技术字段降级**：event count、bundleId 不再抢占主视觉
- **状态可理解**：通过状态标签和次要说明表达“是否连续专注 / 是否高频切换”，而不只是列出 App 名
