# PRD：四期 — 体验优化与分组管理（V4）

**版本**：v1.0 · 2026-04-05  
**上游基准**：`prd/PRD_三期_用户场景与正式壳.md`（V3 正式壳与四模块主导航）  
**关联实现**：桌面端 Tauri + React（`project/`）

---

## 1. 背景与问题

三期交付了「今日透视 / 时间线 / 日报告 / 设置」四模块壳与开发模式下的工具入口，但在日常使用中暴露出三类摩擦：

1. **采集状态不可见**：用户难以在主导航页面一眼判断当前是否在采集，需进入开发工具中的「会话」等页才能操作开始/停止。
2. **应用分组（Intent）门槛高**：分组入口藏在开发工具中，与「时间线 / 透视」对事项类型的依赖不匹配；预设少、无批量能力，多应用调整需重复逐条保存。
3. **冷启动无默认建议**：未配置映射时，新会话的 Intent 依赖用户手动维护，缺少对常见桌面应用的合理初始分类。

本迭代在**不改变一期采集语义、不突破离线优先默认**的前提下，补齐上述体验与效率问题。

## 3. 用户与场景

### 3.1 主路径

- **看一眼就能判断是否在采集**：在任意主导航页面顶栏即可看到状态，并可单击切换开/停。
- **按应用管理时间线中的「事项类型」**：从侧栏「应用分组」进入，列表展示历史出现过的应用键（应用名 + Bundle），可筛选、可批量改分组。
- **信任系统默认、必要时再覆盖**：常见 IDE / 浏览器 / IM 等自动带有内置建议；用户手动规则始终优先于内置。

### 3.2 边缘与维护场景

- **批量统一分组**：多选应用 → 选择目标分组（或自定义文案）→ 一次应用到选中项。
- **历史数据仅有会话、尚未写入 Intent 字段**：提供「补齐历史未分组会话」操作，仅填充当前仍为空的 `intent`，不覆盖已有值。
- **升级应用后词表更新**：内置映射随应用版本携带的目录版本号刷新（实现层：`intent_builtin_catalog_version`）。

## 5. 功能与能力清单

### 5.1 P0（本迭代必达）

| 能力 | 说明 |
|------|------|
| 顶栏采集状态与切换 | 全主导航页可见；状态与后端 `tracking` 及事件 `tracking_state_changed` 一致 |
| 启动默认开始采集 | 桌面端（macOS / Windows）进程启动后 `tracking` 默认为开；无采集能力平台保持关 |
| 应用分组主导航入口 | `/intents` 出现在「功能」分组，**不依赖**开发模式 |
| 内置 Intent 词表 | 常见应用 `app_name` / `bundle_id` 精确匹配 → 三类预设（编码开发 / 研究检索 / 通讯沟通）；存 `intent_mapping.is_builtin=1`，优先级低于用户规则 |
| 词表版本与升级 | `settings.intent_builtin_catalog_version`；版本 bump 时删除旧内置行并重灌 |
| 列表展示来源 | 每应用键展示 `intentSource`：`none` / `builtin` / `user` |
| 批量设置分组 | 多选 + 统一 Intent（含清空为未分类、自定义文案）→ `set_intent_for_app_aggregates_batch` |
| 历史补齐 | `backfill_session_intents_from_mappings`：仅更新 `intent` 为空的会话 |
| 单行快速保存 | 保留「仅此项」与映射 + 全量会话同步语义（同三期前 `set_intent_for_app_aggregate`） |

### 5.2 P1（本迭代已含或随实现微调）

| 能力 | 说明 |
|------|------|
| 筛选与搜索 | 来源分段（全部 / 未映射 / 内置 / 手动）+ 搜索应用名、Bundle、当前分组 |
| 统计摘要 | 列表头展示应用键总数及内置 / 手动 / 未映射计数 |
| 顶栏启动事件 | 若默认采集中，启动后 emit `tracking_state_changed`，避免首帧 UI 闪烁 |

### 5.3 明确不做（本迭代）

| 项目 | 说明 |
|------|------|
| 运行时联网拉取词表 | 内置字典为**打包在客户端**的静态目录；远程更新可列为后续迭代 |
| 模糊匹配 / 正则意图 | 仍为精确匹配；避免误伤应用名 |
| 修改时间线配色语义 | 仍沿用既有 Intent 字符串与时间线 bucket 逻辑 |

## 6. 数据与接口（摘要）

### 6.1 表与规则

- `intent_mapping`：新增/刷新内置行（`is_builtin=1`）；用户规则仍为 `is_builtin=0`，优先级高于内置。
- `window_sessions.intent`：批量与补齐仅按 PRD 约定更新；聚合管道新建会话时仍 `resolve_intent`（含内置）。

### 6.2 Tauri 命令（增量或语义延伸）

| 命令 | 作用 |
|------|------|
| `list_app_intent_aggregates` | 增加返回字段 `intentSource`（camelCase） |
| `set_intent_for_app_aggregates_batch` | 批量写入用户规则并更新对应会话 `intent` |
| `backfill_session_intents_from_mappings` | 返回更新的会话行数 |

既有命令：`start_tracking`、`stop_tracking`、`is_tracking`、`set_intent_for_app_aggregate` 等保持不变。

### 6.3 前端模块

- `AppShell.tsx`：主导航项、顶栏采集状态、三期标题逻辑扩展。
- `IntentManagePage.tsx`（路由仍为 `/intents`）：批量条、筛选、列表与 Virtuoso。
- `intentPresets.ts`：预设分组常量与来源文案，与报告/透视解析对齐。

## 7. 交互与文案原则

- **采集**：状态文案固定为「采集中」「停止采集」；按钮需可访问性属性（如 `aria-pressed`）与简要 `title`。
- **分组页**：页面标题对用户展示「应用分组」；说明中强调「保存后同步该应用下全部历史会话」。
- **内置 vs 手动**：列表中以徽章区分「内置」「手动」「未映射」，避免用户误以为无法覆盖内置结果。
- **危险操作**：历史补齐前须确认对话框，明确「不覆盖已有 Intent」。

## 8. 验收要点（Given / When / Then 摘要）

1. **主导航**：关闭开发模式时仍可见「应用分组」并可进入；开发工具中不再出现重复 Intent 入口。
2. **采集**：顶栏状态与托盘/会话页逻辑一致；冷启动默认采集中（macOS/Windows）；停止后新前台会话仍按既有规则聚合，但 tracker 尊重停止状态。
3. **内置词表**：安装内置应用键（如 VS Code、Chrome、微信）时，新会话 Intent 或列表「当前分组」能体现内置建议；用户保存该应用后显示为「手动」且结果以用户为准。
4. **批量**：选中 ≥2 个应用键，一次应用同一分组后，列表与 `window_sessions` 对应行一致（抽样 SQL 或会话页核对）。
5. **补齐**：仅 `intent` 为空的会话被写入；已有值不变。
6. **构建**：`npm run build` 通过；`cargo test`（含 `intent_mapping`）通过。
