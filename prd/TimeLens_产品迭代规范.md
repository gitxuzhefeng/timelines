# TimeLens 产品迭代规范

> **目的**：把历次迭代中验证有效的做法抽象为**可重复、可裁剪**的流程，使任意阶段在「产品设计 → 技术方案 → 任务与排期 → 研发测试 → 运维交付 → 复盘」上步骤一致、责任清晰、可追溯到验收。  
> **适用**：major / minor 功能迭代与里程碑发布；**热修与救火**可裁剪部分环节，但须在工单或发布说明中**书面记录裁剪项与风险**。  
> **原则**：本规范描述**角色、闸门、工件与完成定义**；具体技术栈、目录命名、测试命令以**当期架构文档与仓库约定**为准。  
> **版本**：v2.2 · 2026-04-05

---

## 1. 文档体系与角色

### 1.1 标准工件（抽象命名）

下列工件为**推荐全集**；单期迭代可按复杂度裁剪，但裁剪项须在 PRD 或里程碑计划中注明。

| 工件类型 | 命名模式（建议） | 维护责任 | 用途 |
|----------|------------------|----------|------|
| 产品设计（PRD） | `PRD_<阶段>_<主题>.md` | 产品 Owner | 范围、边界、验收叙事、与路线图对齐 |
| 技术架构 / 技术方案 | 由仓库约定（如 `rule/*TechArch*.md`） | 技术 Owner | 分层、模块、数据与接口契约、非功能需求、禁止项 |
| 里程碑与研发验收 | `<阶段>_里程碑与验收计划.md` 或 `<阶段>_研发任务与验收计划_P*.md` | 技术 Owner + 测试 | 模块依赖、Given/When/Then、排期与联调点 |
| 领域精确定义（按需） | 如 `<阶段>_分析指标字典.md`、API 契约包 | 技术 Owner + 产品 | 指标/SQL/字段语义与 PRD 一致，避免实现漂移 |
| 测试与签收 | `docs/` 下按里程碑命名，或与验收计划合并 | 测试 | 自动化映射、手动 E2E、性能基线、发布前 checklist |
| 用户向说明 | 用户手册、入门引导、FAQ | 产品 / 研发 | 对外行为与已知限制一致 |
| 构建与发布 | 打包指南、CI 配置、`scripts/*` | 研发 | 版本号、产物、可重现构建 |

**可追溯性要求**：P0 需求条目应能指到至少一处——架构中的模块/接口/表，或验收表中的 Given/When/Then；避免出现「PRD 有、架构无、测试无」的孤儿需求。

### 1.2 关键角色（可一人兼任）

| 角色 | 职责摘要 |
|------|----------|
| **产品 Owner** | 范围冻结与变更裁决、验收拍板、用户文档与 Release 叙事 |
| **技术 Owner** | 架构与实现边界、模块拆分、技术风险与降级策略 |
| **接口契约 Owner** | 命令/事件/表结构/前后端类型命名一致；变更同步 PRD、架构、任务表与测试对照 |
| **测试** | 用例与验收勾选、回归范围、缺陷分级；不写业务代码 |

### 1.3 TimeLens 仓库映射（示例）

| 抽象工件 | 一期参考 | 二期参考 |
|----------|----------|----------|
| PRD | `prd/PRD_一期_数据底座.md` | `prd/PRD_二期_智能洞察.md`（及二期截图增强等拆分 PRD） |
| PRD（三期） | — | `prd/PRD_三期_用户场景与正式壳.md` |
| PRD（四期） | — | `prd/PRD_四期_体验优化与分组管理.md`（三期壳之后的体验与分组迭代） |
| 架构 | `rule/TimeLens_TechArch_V2.md`（一期基线） | **增量**：`rule/TimeLens_TechArch_Phase2.md`（二期在 v2.2 之上扩展）；基线仍以 V2 为准 |
| 里程碑与验收 | `docs/archived/一期/一期_研发任务与验收计划_P0.md`（历史路径） | `prd/二期_里程碑与验收计划.md` + `prd/二期_产品研发任务计划表.md`（执行主计划） |
| 领域字典（按需） | — | `prd/二期_分析指标字典.md` |
| 测试说明 | `docs/archived/一期/M9_验收与测试.md` 等 | 按二期里程碑在 `docs/` 或验收计划中维护 |

## 2. 迭代阶段闸门（Stage Gates）

每个里程碑建议按顺序通过下列闸门；**未通过不得进入下一阶段**（紧急修复除外，须备案）。

| 阶段 | 准入条件 | 产出物 |
|------|----------|--------|
| **G0 立项** | 路线图或版本目标明确；本迭代一句话可复述 | 迭代目标说明（issue、PRD 修订记录或简短立项文档） |
| **G1 需求冻结** | PRD 含：目标用户、必做/不做、优先级分级、与上游/上期的衔接与已知限制 | 带版本号与日期的 PRD |
| **G2 方案对齐** | 技术方案与 PRD 的 P0（或当期必交付）一致；依赖、禁止项、非功能需求无冲突 | 架构/方案文档版本更新；必要时「愿景 vs 本迭代落地」对照表 |
| **G3 任务与排期** | 模块依赖关系已定；验收条目可测试；接口契约 Owner 已指定 | 里程碑/研发验收计划定稿 |
| **G4 开发完成** | P0 联调通过；**项目约定**的单测/集测全绿；无未关闭 P0 缺陷 | MR/PR 通过评审；可构建的测试产物 |
| **G5 测试签收** | 自动化 + 约定手动 E2E 完成；若当期要求性能/稳定性，基线或抽样已记录 | 勾选后的验收表导出或截图归档 |
| **G6 发布** | 版本号与构建产物规则一致；Release Note + 已知限制；可重现构建 | Tag、发布渠道更新、文档索引更新（按需） |

## 3. 产品设计规范

### 3.1 PRD 必备章节（当期范围）

1. **背景与问题**：解决什么、不解决什么。  
2. **本迭代使命**：在整体路线图中的位置（可与多期对照表一并给出）。  
3. **用户与场景**：主路径与可选边缘场景。  
4. **功能与能力清单**：按 **P0 / P1 / 明确延后** 分级；**「本迭代不做」** 单独成表。  
5. **边界与约束**：产品、交互、合规/隐私、性能等分层写清。  
6. **验收原则**：与里程碑验收表中的 Given/When/Then **同源或可映射**，避免两套标准。  
7. **（按需）降级与可选能力**：如多数据源、可选模块、opt-in 能力（例如 AI）：须写清**默认行为**与**关闭/失败时**用户体验。

### 3.2 防范围蔓延

- 新增需求默认进入 **下一迭代** 或 **P1 池**；进入本迭代 P0 须产品 Owner 批准（工单或书面记录）并 ** bump PRD 版本号**。  
- PRD 中的长期愿景须附 **「本迭代落地子集」**，避免将愿景误读为当期交付。  
- 涉及**隐私、外网、AI、付费或第三方服务**的扩展，须同步更新架构与用户可见说明，必要时单独立项过闸门。

### 3.3 版本与变更记录

- PRD 文首维护：**版本号、日期、上游基准、取代/关联文档**。  
- 重大范围变更：在 **修订记录** 中写清日期与摘要。

## 4. 技术方案与架构规范

### 4.1 与 PRD 的关系

- 技术方案/架构文档是 **实现层基准**：PRD 的 P0 须有对应模块、数据对象、接口或明确「当期不实现但预留」。  
- 架构中的 **禁止项或假设**（如离线优先、零外网默认）若需突破，须走变更：**PRD + 架构 + 合规/用户说明** 一致更新。

### 4.2 模块与风险分级

- **P0**：核心链路；缺失即迭代失败。  
- **P1**：增强能力；须 **可独立关闭或降级** 且 **核心链路不依赖其始终可用**（除非 PRD 明确）。  
- **延后**：仅保留接口或 Schema 扩展点，不写完整业务逻辑（除非单独立项）。

### 4.3 工程原则（写入架构文档，按项目裁剪）

下列为**常见**原则，具体条目以 `TimeLens_TechArch` 等文档为准；新阶段可增补，不必重复抄入本规范。

- **单写者与一致性**：关键持久化路径避免多写者竞争；迁移与聚合幂等。  
- **分层与语义稳定**：原始数据、聚合层、展示/报告层边界清晰，语义在架构或指标字典中可追溯。  
- **隐私与安全默认值**：采集、存储、外传与 PRD 一致并可测。  
- **AI 与外部服务（若存在）**：默认关闭或可感知授权；失败时回退到非 AI 基线路径（与二期 PRD 一致时可单独成章）。

### 4.4 接口与数据变更控制

- **公共契约**（命令、事件、表结构、共享 DTO）变更须经接口契约 Owner 同步实现、调用方类型与 **验收/测试对照表**。  
- 合并前建议 checklist：**契约三处（或 N 处）已对齐**。

## 5. 任务排期规范

### 5.1 依赖图优先于日历

- 先明确 **模块/Story 依赖**（DAG），再落到周计划；**硬依赖**不可颠倒。  
- 排期表上的「示例周次」仅为沟通用途，**以依赖与风险为准**调整。

### 5.2 任务表结构

- 按模块或里程碑拆分；每块包含：**目标、角色分工、Given/When/Then 验收表**。  
- **联调点**、**环境依赖**（权限、多进程、外部服务）单独标出。

### 5.3 完成定义（Definition of Done）

- 代码已合并默认分支并通过评审。  
- **自动化测试**：按项目约定补充或全绿。  
- 接口与类型已与契约 Owner 对齐。  
- 用户可见行为与 **用户说明 / 已知限制** 一致或已更新文档。

### 5.4 缓冲

- 含 **系统权限、原生 UI、IPC、多端** 的迭代须预留 **联调 + 缺陷修复** 缓冲；避免把乐观日历当作对外承诺。

## 6. 研发与测试规范

### 6.1 自动化与手动的分界

- **优先自动化**：纯逻辑、数据迁移幂等、事务与聚合规则、可稳定映射到需求的单元/集成测试。  
- **默认可手动**：强依赖 **本机权限、GUI、多应用协作** 的路径；须写入手动 E2E 步骤（环境、步骤、预期）。  
- **具体命令**（如 `cargo test`、`pnpm test`）以仓库 README 与 CI 为准，本规范不绑定单一栈。

### 6.2 测试文档要求

- 维护 **「测试用例 ↔ PRD / 验收条目」** 对照或可追溯 ID。  
- **发布前**：至少 **全量自动化（若已有）+ 主路径手动 E2E**；触及数据层/迁移/核心 Writer 的改动须跑约定测试套件。

### 6.3 性能与稳定性

- 当期若声明性能目标：使用**统一记录模板**（日期、构建类型、环境、指标、备注）。  
- 相对架构或 PRD 中的目标偏离时，须在备注说明原因。

### 6.4 发布前 Checklist（建议每次发版）

- [ ] 公共契约与各端封装一致。  
- [ ] 与 PRD 冲突项已落地或已书面备案。  
- [ ] **已知限制** 已写入 Release 或用户文档。  
- [ ] P0 验收已勾选并有归档（截图或导出）。

## 7. 运维与交付规范

### 7.1 版本号

- **唯一来源**：由项目约定（如桌面端的 `tauri.conf.json`、前端的 `package.json`、服务的 tag 策略）；**脚本化 bump**，避免多处手写不一致。

### 7.2 构建与产物

- 发布命令、产物命名、签名/notarization（若有）须在 **打包/部署文档** 与脚本中可查；新成员依赖项保持更新。

### 7.3 Release Note

- 建议包含：**新增 / 修复 / 破坏性变更 / 已知限制 / 升级与迁移说明**。  
- 区分 **「本迭代不做」**（规划）与 **「已知限制」**（当前版本客观约束）。

### 7.4 自动更新、遥测与第三方服务

- 默认 **不启用** 自动更新、遥测或不可控第三方，除非 PRD 与架构已纳入且用户可感知同意；启用前建议单独立项评审。

---

## 8. 复盘与持续改进

- **每个里程碑结束后** 短复盘（可与 G6 合并）：对照本规范检查闸门执行情况、被跳过项及原因。  
- 更新本规范：在 **修订记录** 中增加版本、日期与摘要；重大流程变更通知干系人。

---

## 9. 规范本身的分层

| 层级 | 内容 | 变更频率 |
|------|------|----------|
| **L1 本规范** | 闸门、角色、工件类型、DoD、通用原则 | 低；跨阶段复用 |
| **L2 架构 / 指标字典** | 技术细节、禁止项、语义与 SQL | 中；随版本演进 |
| **L3 当期 PRD + 里程碑计划** | 具体范围与验收表 | 高；每迭代更新 |
