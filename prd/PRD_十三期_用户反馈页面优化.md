# PRD：十三期 — 基于用户反馈的页面体验优化（V13）

> 说明：本期优化来源于真实用户反馈，并非「应用内用户意见/工单反馈页」。若对仓库内文件名有引用，可与本标题同义理解。

**版本**：v1.1 · 2026-04-23
**上游基准**：`prd/PRD_十二期_页面体验优化.md`
**关联实现**：桌面端 Tauri + React（`project/`）
**遵循规范**：`prd/TimeLens_产品迭代规范.md`

## 1. 背景与问题（PRD_BACKGROUND）

十二期完成了页面信息架构的首轮优化（今日透视场景化、时间线分组与倒序、设置分层等）。但经过一段时间的真实使用，用户反馈集中暴露了以下六大体验痛点：

### 1.1 时间线页面

- **应用占比信息不足**：当前仅展示 Top 3 应用的 chip，用户无法了解完整的应用使用分布
- **时间轴颗粒度过细**：每一次窗口切换都独立展示，一天下来上百条记录，常用应用（通常不超过 10 个）大量重复出现，信息噪声极高
- **缺少概括视角**：十二期虽有分组逻辑（相邻同应用 3 分钟内合并），但合并力度不够，用户仍无法快速回答「8:00-9:00 我主要在干什么」

### 1.2 今日透视（TodayLens）

- **内容单调缺乏吸引力**：每次打开看到的内容差异不大，用户失去回访动机
- **日报告纯文本呈现**：总览、时间去向、注意力、打断与通知、输入节奏等数据板块全部以文本形式展示，信息密度低且不直观
- **数据不实时**：用户使用了一段时间后页面仍停留在旧数据，需手动点击刷新

### 1.3 周报告

- **热力图占比过大**：24×7 的热力图在页面中占据过多空间，与 GitHub 贡献热力图的紧凑风格差距明显
- **AI 叙事仅 Markdown**：纯文本叙事不够直观，用户希望有图表化的周度总览

### 1.4 应用分组（Intent 管理）

- **分组不可自定义**：当前仅支持预设的 4 类意图（深度工作/协作沟通/浏览与学习/其他），用户无法按自己的工作场景创建分组
- **缺少智能匹配**：每个应用都需手动逐一分配意图，效率低下

### 1.5 OCR 检索

- **功能使用率极低**：OCR 检索作为独立导航菜单项，实际几乎无人使用，占据了宝贵的导航空间

## 3. 用户与场景（PRD_USERS）

### 3.1 核心用户画像

- **效率回顾型用户**：每天查看时间线，想快速了解「今天时间花在哪了」，不想逐条翻阅
- **数据可视化偏好用户**：相比文字更喜欢图表，希望一眼看到趋势和分布
- **个性化管理用户**：工作场景多样（开发、设计、运营等），需要按自己的分类体系管理应用

### 3.2 主路径场景

| 场景 | 用户行为 | 期望体验 |
|------|----------|----------|
| 快速回顾 | 打开时间线，想知道上午主要做了什么 | 看到「8:00-10:00 主要在 VS Code」而非 50 条切换记录 |
| 应用分布 | 想了解今天各应用的使用占比 | 展开完整应用列表，看到每个应用的时长和百分比 |
| 日报浏览 | 打开今日透视查看日报 | 图表化展示各维度数据，切换到文本视图可复制 |
| 周度总结 | 周一查看上周报告 | 紧凑热力图 + 图表化周报，一屏内看完关键信息 |
| 分组管理 | 新安装了几个应用，想归类 | 点击「智能匹配」自动分组，仅需确认少量未识别项 |
| 实时数据 | 使用中切换到 TimeLens 查看 | 页面自动刷新，无需手动操作即可看到最新数据 |

### 3.3 边界（PRD_SCOPE）

**不做 / 延后**

- 本期不做时间线的跨天合并视图
- 不做日报告的导出为图片功能（可后续迭代）
- 智能匹配的 AI 模式依赖用户已配置 AI 渠道，未配置时仅使用规则匹配

**采集与持久化（澄清）**

- **不改变**：原始 WindowSession 等**采集管道**与落库字段语义（无新采集探针、不改原始事件结构）。
- **允许**：展示层聚合（时间线概括算法）、**意图配置**（`custom_intents`、resolved intent 写入）、以及为本期 UI 服务的**只读聚合 API**（如 `get_all_app_usage_stats`）。不得将「仅展示层」理解为「前端-only、后端零变更」。

**自动刷新范围（本期明确）**

- P0 仅对 **今日透视**、**时间线** 两页：路由切入与窗口聚焦时触发自动刷新（见 4.1.4）。**日报告 / 周报告 / 应用分组** 本期不强制纳入同一套自动刷新（用户可仍用手动刷新或既有行为）。

**日界与时区**

- 「当日」会话边界、统计口径与 **App 现有「一天」定义**一致（含跨午夜归属）；若产品全局无时区设置，以系统本地时区为准。

### 3.4 失败与降级（PRD_DEGRADE）

| 场景 | 行为 |
|------|------|
| 日报告「输入节奏」无剪贴板聚合数据 | 隐藏该图表区块或展示空状态文案（可切换文本视图查看）；不得因缺数据阻塞整页 |
| 图表库或单图渲染失败 | 该区块错误边界 + 提示；文本/Markdown 视图可正常访问 |
| 智能匹配 AI 超时 / 额度 / 未配置 | 回退规则匹配；按钮在未配置 AI 时仍可仅用规则，或明确禁用「AI 增强」子选项并提示 |
| localStorage 不可用 | 视图偏好无法持久化时降级为默认视图，不阻断主流程 |
| 外向 AI 请求 | 仅发送匹配所需最小字段；需在设置或首次使用路径具备**隐私说明**（会话标题可能含敏感信息），用户继续使用即视为知晓 |

**4.1.2.1 概括视图合并算法**

```
输入：当日所有 WindowSession，按 startTime 升序排列
输出：合并后的时段列表，每个时段包含 { startTime, endTime, primaryApp, secondaryApps[], totalDurationMs }

规则：
1. 窗口推进（非墙钟对齐）：
   - 从当日第一条会话的 startTime 起，取连续时间轴上长度 30 分钟的区间作为「基础窗口」。
   - 下一窗口从上一窗口的结束时刻起算，同样 30 分钟，直至覆盖当日最后一条会话的结束时间。
2. 窗口内统计：在窗口时间范围内，按应用累计使用时长（含 <1min 会话，见规则 6）。**占比分母** = 窗口内各应用时长之和（即落在该窗口内的会话并集时长，不大于 30 分钟墙钟）。
3. 主应用判定：
   - 若某应用占上述分母 ≥ 60%，该应用为主应用，该窗口输出一条候选时段。
   - 若无单一应用 ≥ 60%，取 Top 2 应用作为联合主展示；primary 为时长第一，secondary 仅保留第二（不在一行里堆更多应用）。
4. 窗口延伸：若**相邻**基础窗口的「主展示」相同（联合展示时需两组应用集合一致），则合并为一条时段；合并后时段长度最长不超过 2 小时；超过则拆分回多条连续时段。
5. 展示用次要应用：主应用占比文案中的「也用了 A、B…」仅展示除主应用外时长最高的至多 2 个应用（按降序），与规则 3 的 Top 2 判定独立。
6. 短会话吸收：< 1 分钟的会话不单独成条展示，但必须计入所属窗口内各应用时长统计。
7. 展示格式示例：「8:00 - 9:30  VS Code（78%）· 也用了 Chrome、Terminal」
```

#### 4.1.3 日报告 — 图表视图

| 能力 | 说明 |

#### 4.2.4 应用分组 — 智能匹配

| 能力 | 说明 |
|------|------|
| 智能匹配按钮 | 应用分组页面顶部增加「智能匹配」按钮 |
| 规则匹配（默认） | 基于应用名称和会话标题的关键词匹配，将应用自动归入最匹配的分组 |

**4.2.4.1 规则匹配逻辑**

```
输入：未映射应用列表、所有分组（预设 + 自定义）及其名称
输出：每个应用的建议分组

规则优先级：
1. 分组关键词表：维护「分组 → 关键词列表」（含预设分组别名，如深度工作 ← VS Code、Xcode、Terminal…），应用名或 bundleId 命中则建议该分组；**不**依赖「应用名包含分组中文名」字面匹配
2. 类别匹配：内置应用类别映射表（IDE → 深度工作，浏览器 → 浏览与学习，IM → 协作沟通…）
3. 会话标题分析：该应用最近 7 天会话标题的高频词与分组关键词表交集命中
4. 未匹配：标记为「建议人工分配」
```

**4.2.4.2 AI 匹配 Prompt 设计**

```
系统提示：你是一个应用分类助手。根据以下应用信息，将每个应用分配到最合适的分组中。

输入数据：
- 可用分组列表：[{name, description}]
- 待分类应用：[{appName, bundleId, recentSessionTitles[]}]

输出格式：JSON 数组 [{appName, bundleId, suggestedGroup, confidence}]

要求：
- confidence 为 high/medium/low
- 无法确定的应用 suggestedGroup 设为 null
```

## 6. 数据模型变更（PRD_DATA）

### 6.1 新增表：`custom_intents`

```sql
CREATE TABLE IF NOT EXISTS custom_intents (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,
    color       TEXT,           -- HEX 色值，如 #3B82F6
    sort_order  INTEGER DEFAULT 0,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 6.2 `app_intent_aggregates` 表变更

现有 `resolved_intent` 字段兼容自定义分组名称，无需改表结构。自定义分组名称直接作为 intent 值写入。

## 11. 验收标准（PRD_ACCEPTANCE）

### P0 验收

- [ ] 时间线页面 Top 3 chip 下方可展开查看所有应用的时长与占比
- [ ] 时间线默认展示概括视图，30 分钟窗口合并，主应用占比 ≥ 60% 时合并为单条
- [ ] 可切换到完整视图，展示与当前版本一致的逐条记录
- [ ] 视图偏好在 localStorage 中持久化
- [ ] 日报告默认展示图表视图，包含总览卡片、时间去向环形图、注意力时间轴、打断柱状图；输入节奏图在**有数据**时展示，无数据时符合 3.4 降级
- [ ] 日报告可切换到文本视图，内容与当前版本一致
- [ ] 切换页面到今日透视/时间线时自动刷新数据（其他页面本期不要求）
- [ ] TimeLens 窗口从后台恢复且当前为今日透视或时间线时自动刷新数据
- [ ] 自动刷新间隔不小于 30 秒
- [ ] OCR 检索已从主导航移入开发工具子菜单

### P1 验收

- [ ] 周报告热力图高度 ≤ 200px，格子尺寸 10-12px，5 级色阶
- [ ] 周报告默认展示图表视图，含总览卡片、应用趋势图、评分走势图
- [ ] 可切换到 Markdown 视图
- [ ] 应用分组页面支持创建/编辑/删除自定义分组
- [ ] 应用分组下拉选择器包含自定义分组选项
- [ ] 页面默认筛选显示「未映射」应用
- [ ] 智能匹配（规则模式）可自动为未映射应用建议分组
- [ ] AI 匹配模式在已配置 AI 渠道时可用
- [ ] 匹配结果先预览后确认，不自动覆盖

### 通用验收

- [ ] 所有新增文本均有 zh-CN 和 en 翻译
- [ ] 深色/浅色主题下图表配色正确；关键序列不仅依赖颜色区分（图例/形态/标签可辨）
- [ ] 页面无明显性能退化（开发机参考：时间线 **500+** 条当日会话时，概括视图前端合并 + 首屏渲染 **< 200ms**，Chrome 性能面板或等价方式记录）
