# TimeLens 二期研发任务与验收计划

> **范围**：二期全部里程碑（M0–M5），覆盖 P0 交付物；P1 增强项单独标注，不阻塞主线验收。  
> **用途**：研发分工、排期、依赖对齐；测试同事按 **Given / When / Then** 勾选验收。  
> **版本**：v1.0 · 2026-03-31  
> **配套文档**：`PRD_二期_智能洞察.md` v3.1、`二期_分析指标字典.md` v1.0、`二期_产品研发任务计划表.md` v1.0（开发执行主计划与工作包）

---

## 一、角色说明

| 角色 | 职责摘要 |
| --- | --- |
| **后端** | Rust/Tauri：补齐采集引擎、分析聚合引擎、报告生成器、AI 调用层、新增 IPC 接口 |

## 二、模块依赖总览

```
M0 数据管道审计与补齐
  │
  ├─────────────────────┐
  ▼                     ▼
M1 分析引擎 v1         M2 分析引擎 v2
 (指标组 A+B)           (指标组 C+D)
  │                     │
  └──────────┬──────────┘
             ▼
M3 日终复盘报告 + 导出
  │
  ├─────────────────────┐
  ▼                     ▼
M4 最小可用 UI          M5 AI 叙事层 + 纠错
  │                     │
  └──────────┬──────────┘
             ▼
         二期发布
```

**硬依赖（不可颠倒）**

| 后置模块 | 依赖前置 | 原因 |
| --- | --- | --- |
| M1 | M0 | 分析需要数据表就绪且有数据 |
| M2 | M0 | 同上 |
| M3 | M1 + M2 | 报告需要全部 P0 指标就绪 |
| M4 | M3 | UI 需要报告数据可渲染 |
| M5 | M3 + M4 | AI 消费报告结果；纠错需要 UI |

**可并行**

| 并行组 | 说明 |
| --- | --- |
| M1 ∥ M2 | 指标组 A+B 和 C+D 无数据依赖 |
| M4 骨架 ∥ M3 | UI 页面框架可在 M1 完成后开始搭建，M3 完成后填充真实数据 |

### M0 · 数据管道审计与补齐

**目标**：确保一期 12 张表全部存在且持续产生数据；6 大采集引擎全部运行；实现 `get_pipeline_health` 接口。

| 角色 | 任务 |
| --- | --- |
| **后端** | 审计 12 张表状态；补齐 Input Dynamics / Clipboard Flow / Context Switch（若一期未完成）；实现 Notification Tracker + Ambient Context；实现 `get_pipeline_health` Command |
| **前端** | 无（健康度面板在 M4） |
| **测试** | 逐表检查数据完整性；权限降级路径 |

#### 验收条目

- [ ] **M0-01 · 表结构完整性**
  - **Given** 应用首次启动（或从一期升级）
  - **When** 数据库初始化完成
  - **Then** 14 张表全部存在（一期 12 张 + 二期新增 `daily_analysis` + `daily_reports`）；`schema_migrations` 记录正确版本

- [ ] **M0-02 · P0 引擎数据持续性**
  - **Given** 已授权必要权限，采集运行中
  - **When** 运行 30 分钟以上
  - **Then** `raw_events`、`window_sessions`、`snapshots`、`app_switches` 均有当日数据且 `timestamp_ms` 合理递增

- [ ] **M0-03 · P1/延后引擎数据持续性**
  - **Given** 已授权必要权限，采集运行中
  - **When** 运行 30 分钟以上
  - **Then** `input_metrics`（每 5 秒一条）、`clipboard_flows`（有 copy/paste 操作时）均有数据

- [ ] **M0-04 · Notification Tracker**
  - **Given** 采集运行中，系统通知到达（如微信消息）
  - **When** 通知弹出
  - **Then** `notifications` 表有新记录，`source_app` 正确，`caused_switch` 逻辑正确

- [ ] **M0-05 · Ambient Context Engine**
  - **Given** 采集运行中
  - **When** 每 30 秒轮询
  - **Then** `ambient_context` 表有新记录；WiFi SSID、显示器数量、电池状态等字段非全 NULL

- [ ] **M0-06 · 健康度接口**
  - **Given** 采集运行中
  - **When** 调用 `get_pipeline_health`
  - **Then** 返回 JSON 包含 6 个引擎的状态（`running` / `degraded` / `stopped`）、各引擎最近数据时间戳、异常计数

- [ ] **M0-07 · 权限缺失降级**
  - **Given** 未授权辅助功能
  - **When** 启动采集
  - **Then** Tracker/Capture 标记为 `degraded`；应用不崩溃；`get_pipeline_health` 正确反映状态

- [ ] **M0-08 · Notification 权限降级**
  - **Given** 系统未允许通知监听（或 macOS 限制）
  - **When** Notification Tracker 尝试监听
  - **Then** 引擎标记为 `degraded`；其他引擎不受影响

### M1 · 分析引擎 v1（指标组 A + B）

**目标**：实现时间分布（A1–A4）和注意力切换（B1–B4）指标的计算；结果写入 `daily_analysis` 表；全部指标可通过 fixture 数据单测。

| 角色 | 任务 |
| --- | --- |
| **后端** | `analysis/engine.rs`：指标组 A、B 计算逻辑；`daily_analysis` 表写入；`generate_daily_analysis` Command（部分）；深度工作段算法 |
| **前端** | 无 |
| **测试** | 准备 fixture 数据集（已知答案的 window_sessions + app_switches）；golden test |

#### Fixture 数据集要求

测试必须准备一份**标准 fixture**——已知答案的数据，用于验证聚合正确性：

| fixture 内容 | 数量 | 已知答案 |
| --- | --- | --- |
| `window_sessions` | 50 条，覆盖 3 个 intent + NULL | 各 intent 时长、Top 5 App、总活跃时长 |
| `app_switches` | 80 条，含 1 段 >30 分钟无切换 | 各小时切换数、Top 5 对、深度段起止、碎片化指数 |

#### 验收条目

- [ ] **M1-01 · A1 总活跃时长**
  - **Given** fixture 数据集已导入（总时长已知为 28800000ms = 8h）
  - **When** 调用 `generate_daily_analysis({ date: "fixture-date" })`
  - **Then** `daily_analysis.total_active_ms = 28800000`（允许 ±1000ms 误差）

- [ ] **M1-02 · A2 Intent 分布**
  - **Given** fixture 中"编码开发"= 3h、"研究检索"= 2h、"通讯沟通"= 1.5h、NULL = 1.5h
  - **When** 分析完成
  - **Then** `intent_breakdown` JSON 数值与预期一致；占比计算正确

- [ ] **M1-03 · A3 Top 5 应用**
  - **Given** fixture 数据
  - **When** 分析完成
  - **Then** `top_apps` 列表按时长降序排列，前 5 与手动统计一致

- [ ] **M1-04 · B1 每小时切换次数**
  - **Given** fixture 中 10:00–11:00 有 25 次切换
  - **When** 分析完成
  - **Then** `switches_per_hour` 中 `"10": 25`

- [ ] **M1-05 · B2 Top 5 切换对**
  - **Given** fixture 中 Cursor→Chrome 出现 15 次
  - **When** 分析完成
  - **Then** `top_switch_pairs` 首项为 `{"from": "Cursor", "to": "Chrome", "count": 15}`

- [ ] **M1-06 · B3 深度工作段**
  - **Given** fixture 中 14:00–14:55 无 app_switch，期间 sessions 均为"编码开发"
  - **When** 分析完成
  - **Then** `deep_work_segments` 包含一个 55 分钟的段（start/end 匹配）

- [ ] **M1-07 · B3 深度段 intent 拆分**
  - **Given** 某段无切换 40 分钟，但其中 sessions 在第 20 分钟从"编码开发"变为"研究检索"
  - **When** 分析完成
  - **Then** 拆分为两个段（各约 20 分钟），而非一个 40 分钟段

- [ ] **M1-08 · B4 碎片化指数**
  - **Given** fixture 中共 96 个 5 分钟窗口，其中 24 个窗口内有 ≥3 个不同 to_app
  - **When** 分析完成
  - **Then** `fragmentation_pct = 25.0`

- [ ] **M1-09 · app_switches 降级**
  - **Given** `app_switches` 表为空
  - **When** 调用分析
  - **Then** 指标组 B 全部为 NULL / 默认值；`degraded_sections` 包含 `"app_switches"`

- [ ] **M1-10 · 性能基准**
  - **Given** 模拟一天数据量：500 条 window_sessions + 400 条 app_switches
  - **When** 运行指标组 A+B 聚合
  - **Then** 耗时 ≤2 秒

### M2 · 分析引擎 v2（指标组 C + D）

**目标**：实现打断分析（C1–C4）和输入节奏（D1–D4）指标；完善 `daily_analysis` 写入；降级逻辑全覆盖。

| 角色 | 任务 |
| --- | --- |
| **后端** | `analysis/interruption.rs`：指标组 C；`analysis/input_rhythm.rs`：指标组 D（含心流/挣扎评分）；降级分支 |
| **前端** | 无 |
| **测试** | 扩展 fixture：增加 notifications + input_metrics；降级矩阵 |

#### Fixture 数据集扩展

| fixture 内容 | 数量 | 已知答案 |
| --- | --- | --- |
| `notifications` | 30 条，5 个来源 App | 各 App 通知数、caused_switch 率、avg delay |
| `input_metrics` | 200 条，覆盖高 KPM 和低 KPM 时段 | 按小时 AVG KPM、日均退格率、心流/挣扎分 |

#### 验收条目

- [ ] **M2-01 · C1 打断者排行**
  - **Given** fixture 中微信 23 次通知，其中 15 次 caused_switch
  - **When** 分析完成
  - **Then** `top_interrupters` 首项 `{"app": "微信", "count": 23, "switch_rate": 65.2, ...}`

- [ ] **M2-02 · C3 打断恢复成本**
  - **Given** fixture 中 5 次 notification 切换后均有返回原 App 的记录
  - **When** 分析完成
  - **Then** 恢复成本 AVG 与手动计算一致（允许 ±500ms）

- [ ] **M2-03 · C4 深度工作中打断**
  - **Given** M1 fixture 中深度段 14:00–14:55 内有 2 条 notifications
  - **When** 分析完成
  - **Then** `interrupts_in_deep = 2`

- [ ] **M2-04 · C 降级（无 notifications）**
  - **Given** `notifications` 表为空，但 `app_switches` 中有 `switch_type = 'notification'` 的记录
  - **When** 分析完成
  - **Then** C1 使用降级 SQL；`switch_rate` 字段为 NULL；`degraded_sections` 包含 `"notifications"`

- [ ] **M2-05 · D1 KPM 趋势**
  - **Given** fixture 中 10:00–11:00 的 input_metrics 平均 KPM = 52.3
  - **When** 分析完成
  - **Then** `kpm_by_hour` 中 `"10": 52.3`

- [ ] **M2-06 · D2 退格率**
  - **Given** fixture 中日均 delete_ratio = 0.083
  - **When** 分析完成
  - **Then** `avg_delete_ratio` ≈ 0.083（允许 ±0.001）

- [ ] **M2-07 · D3 心流评分**
  - **Given** fixture 中一条 input_metrics：kpm=80, delete_ratio=0.03, burst=3, interval=5s, session_duration=2400s
  - **When** 逐行计算 flow_score
  - **Then** `flow_score ≈ (0.667*0.3 + 0.9*0.2 + 0.028*0.25 + 0.667*0.25) * 100 ≈ 55`（允许 ±2）

- [ ] **M2-08 · D4 挣扎评分**
  - **Given** fixture 中一条 input_metrics：delete_ratio=0.25, undo_count=4, interval=5s, burst=1, longest_pause=20000ms
  - **When** 逐行计算 struggle_score
  - **Then** 计算结果与公式手算一致（允许 ±2）

- [ ] **M2-09 · D 降级（无 input_metrics）**
  - **Given** `input_metrics` 表为空
  - **When** 分析完成
  - **Then** D1–D4 全部为 NULL；`degraded_sections` 包含 `"input_metrics"`

- [ ] **M2-10 · 全部 P0 指标完整性**
  - **Given** fixture 全量数据（sessions + switches + notifications + input_metrics）
  - **When** 调用 `generate_daily_analysis`
  - **Then** `daily_analysis` 表所有 P0 字段（见指标字典 §十）均有值

### M3 · 日终复盘报告 + 导出

**目标**：基于 `daily_analysis` 生成固定章节结构的 Markdown 报告；写入 `daily_reports` 表；支持导出为 .md 文件。

| 角色 | 任务 |
| --- | --- |
| **后端** | `report/generator.rs`：模板引擎 + 章节拼接；`generate_daily_report` + `export_report` Command；`daily_reports` 表写入 |
| **前端** | 无（UI 在 M4） |
| **测试** | 报告结构验证；数字与 daily_analysis 交叉验证；降级章节标注验证 |

#### 验收条目

- [ ] **M3-01 · 报告章节完整性**
  - **Given** `daily_analysis` 全字段有值
  - **When** 调用 `generate_daily_report({ date, with_ai: false })`
  - **Then** 生成的 Markdown 包含 7 个章节（今日总览、时间去向、注意力切换、打断分析、输入节奏、信息流向、环境上下文）；P1 章节（6、7）若无数据则标注"数据不可用"

- [ ] **M3-02 · 数字溯源一致性**
  - **Given** 已生成报告
  - **When** 逐行核对报告中的数字与 `daily_analysis` 字段
  - **Then** 100% 一致（总活跃时长、深度工作时长、切换次数、碎片化指数、心流评分等）

- [ ] **M3-03 · 降级章节标注**
  - **Given** `daily_analysis.degraded_sections` = `["notifications", "input_metrics"]`
  - **When** 生成报告
  - **Then** 打断分析章节包含"仅基于切换推断"标注；输入节奏章节包含"输入行为数据不可用"标注

- [ ] **M3-04 · 报告存储**
  - **Given** 生成报告成功
  - **When** 查询 `daily_reports` 表
  - **Then** 有一条 `report_type = 'fact_only'` 的记录；`content_md` 非空；`analysis_id` 指向对应 `daily_analysis`

- [ ] **M3-05 · 重新生成**
  - **Given** 已有今日报告
  - **When** 再次调用 `generate_daily_report`
  - **Then** `daily_analysis.version` +1；新报告覆盖旧报告（或新增一条，旧报告保留）

- [ ] **M3-06 · 导出 Markdown**
  - **Given** 已有报告
  - **When** 调用 `export_report({ date, format: "md" })`
  - **Then** 在 `~/.timelens/exports/` 下生成 `timelens-recap-{YYYY-MM-DD}.md` 文件；文件内容与 `content_md` 一致

- [ ] **M3-07 · 无数据日期**
  - **Given** 目标日期无任何 session 数据
  - **When** 调用 `generate_daily_analysis`
  - **Then** `total_active_ms = 0`；报告显示"当天无活跃数据"

- [ ] **M3-08 · 性能**
  - **Given** daily_analysis 全字段有值
  - **When** 生成 Markdown 报告
  - **Then** 耗时 ≤500ms

### M4 · 最小可用 UI

**目标**：实现 4 个页面（复盘页、会话列表、设置页、健康度面板），使非开发者用户可独立完成核心操作流程。

| 角色 | 任务 |
| --- | --- |
| **后端** | 配合字段联调；确保所有 M3 接口稳定 |
| **前端** | 复盘页（Markdown 渲染 + 日期选择 + 生成/导出按钮）；会话列表（虚拟滚动 + 截图联动）；设置页；健康度面板 |
| **测试** | 5 步操作流程端到端验证；长列表性能；无外网请求 |

#### 验收条目

- [ ] **M4-01 · 5 步操作流程**
  - **Given** 应用已采集数据超过 2 小时
  - **When** 非开发者按以下步骤操作
  - **Then** 每步均可成功完成：
    1. 打开应用 → 看到复盘页（如当日已有报告则直接展示）
    2. 点击"生成分析" → 等待指示 → 看到完整 Markdown 报告
    3. 点击"导出" → 文件生成成功
    4. 切换到会话列表 → 点击某会话 → 看到截图
    5. 日期切换 → 能查看历史日报告

- [ ] **M4-02 · 复盘页渲染**
  - **Given** `daily_reports` 中有一份报告
  - **When** 打开复盘页
  - **Then** Markdown 正确渲染为 HTML（标题、表格、列表、引用块均正确）

- [ ] **M4-03 · 会话列表虚拟滚动**
  - **Given** 当日 300+ 条 sessions
  - **When** 滚动列表
  - **Then** 滚动流畅（FPS ≥30），无白屏闪烁

- [ ] **M4-04 · 会话截图联动**
  - **Given** 某 session 有 3 张截图
  - **When** 点击该 session
  - **Then** 展示 3 张缩略图；点击缩略图可放大预览

- [ ] **M4-05 · 设置页**
  - **Given** 打开设置页
  - **When** 查看可配置项
  - **Then** 至少包含：AI 开关（默认关）、采集引擎开关（6 个）、应用黑名单管理

- [ ] **M4-06 · 健康度面板**
  - **Given** 采集运行中
  - **When** 查看健康度
  - **Then** 6 个引擎分别显示绿/黄/红状态；每个引擎显示最近数据时间

- [ ] **M4-07 · 无外网请求**
  - **Given** AI 关闭
  - **When** 使用所有功能
  - **Then** 网络监控无任何外部 HTTP 请求

- [ ] **M4-08 · 分析进行中反馈**
  - **Given** 用户点击"生成分析"
  - **When** 聚合正在运行
  - **Then** UI 显示加载状态；分析完成后自动刷新报告

### M5 · AI 叙事层 + 纠错

**目标**：实现 AI opt-in 报告生成（BYOK）；Intent 纠错入口；应用黑名单。

| 角色 | 任务 |
| --- | --- |
| **后端** | `ai/client.rs`：LLM 调用封装（支持 OpenAI 兼容 API）；`ai/prompt.rs`：固定 prompt 模板；`report/generator.rs` 扩展 AI 路径；纠错接口（`update_session_intent`、`manage_blacklist`） |
| **前端** | 设置页：API Key 输入 + 模型选择 + 隐私说明弹窗；复盘页：AI 增强报告渲染（AI 内容以引用格式区分）；纠错入口（会话列表中修改 intent） |
| **测试** | AI 开/关对比；数字一致性验证；纠错持久化 |

#### 验收条目

- [ ] **M5-01 · AI 首次开启**
  - **Given** AI 开关为关
  - **When** 用户切换为开
  - **Then** 弹出隐私说明弹窗（说明哪些数据会出境、哪些不会）；用户确认后进入 Key 配置

- [ ] **M5-02 · BYOK 配置**
  - **Given** AI 已开启
  - **When** 用户输入 API Key + 选择模型 + 输入 API Base URL
  - **Then** 配置保存到 `settings` 表；Key 不以明文写入日志

- [ ] **M5-03 · AI 增强报告生成**
  - **Given** AI 配置完成，已有 `daily_analysis` 数据
  - **When** 调用 `generate_daily_report({ date, with_ai: true })`
  - **Then** `daily_reports` 新增一条 `report_type = 'ai_enhanced'`；报告中每个章节在事实内容后附 AI 解读段落

- [ ] **M5-04 · AI 数字一致性**
  - **Given** AI 增强报告已生成
  - **When** 核对 AI 文本中的数字
  - **Then** AI 引用的所有数字与 `daily_analysis` 字段值一致（AI 不可编造数字）

- [ ] **M5-05 · AI 关闭后完整性**
  - **Given** AI 已关闭
  - **When** 生成报告
  - **Then** 报告为纯事实版本（`fact_only`），章节完整，无任何"AI 解读"占位符或空白区域

- [ ] **M5-06 · AI 出境内容验证**
  - **Given** AI 调用触发
  - **When** 检查发送给 LLM 的 prompt
  - **Then** 仅包含 `daily_analysis` 聚合数据 JSON + 系统 prompt；不包含原始窗口标题、剪贴板内容、按键数据

- [ ] **M5-07 · AI 调用失败降级**
  - **Given** API Key 无效或网络不通
  - **When** 尝试生成 AI 报告
  - **Then** 返回错误信息；自动回退到 `fact_only` 报告；用户收到明确提示

- [ ] **M5-08 · Intent 纠错**
  - **Given** 某 session 的 intent = "研究检索"
  - **When** 用户在会话列表中将其修改为"编码开发"
  - **Then** `window_sessions.intent` 更新；新规则写入 `intent_mapping`（`is_builtin = 0`）；后续同类 session 自动匹配新规则

- [ ] **M5-09 · 应用黑名单**
  - **Given** 用户将某应用（如"微信"）加入分析黑名单
  - **When** 重新生成分析
  - **Then** 该应用的 sessions 不纳入时间分布、不出现在 Top 应用中

- [ ] **M5-10 · AI 增强与事实报告并存**
  - **Given** 同一日期已有 fact_only 报告
  - **When** 再生成 ai_enhanced 报告
  - **Then** 两份报告共存于 `daily_reports`；UI 可切换查看
