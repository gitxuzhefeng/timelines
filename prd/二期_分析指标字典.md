## 一、通用约定

### 1.1 时间口径

| 约定 | 说明 |
| --- | --- |
| **"一天"** | `00:00:00` ~ `23:59:59`（本地时区），取 `date(timestamp_ms / 1000, 'unixepoch', 'localtime')` |
| **"活跃时段"** | `window_sessions` 中 `duration_ms > 0` 的记录覆盖的时间范围，排除 AFK 间隙 |
| **"小时桶"** | 按 `strftime('%H', timestamp_ms / 1000, 'unixepoch', 'localtime')` 分桶，值域 `00`–`23` |
| **时长显示** | ≥60 分钟显示 `Xh Ym`；<60 分钟显示 `Xm`；<1 分钟显示 `Xs` |

### 1.2 标准化函数

所有需要归一化的评分指标使用统一的 `normalize` 函数：

```
normalize(value, min, max) = clamp((value - min) / (max - min), 0.0, 1.0)

clamp(x, lo, hi) = max(lo, min(hi, x))
```

### 1.3 JSON 字段约定

`daily_analysis` 表中的 JSON 字段统一采用以下格式：

| 类型 | 格式 | 示例 |
| --- | --- | --- |
| 键值映射 | `{"key": value, ...}` | `{"编码开发": 7200000, "研究检索": 3600000}` |
| 排行列表 | `[{"字段1": 值, "字段2": 值}, ...]`，按权重/数量降序 | `[{"app": "Cursor", "ms": 5400000}, ...]` |
| 时间段列表 | `[{"start_ms": N, "end_ms": N, ...}, ...]`，按时间升序 | 深度工作段 |

## 二、指标组 A：时间分布

**数据源**：`window_sessions`  
**优先级**：P0  
**里程碑**：M1

### A1 · 总活跃时长

| 属性 | 值 |
| --- | --- |
| **ID** | `A1_total_active_ms` |
| **含义** | 当天所有非 AFK 会话的时长总和 |
| **存入字段** | `daily_analysis.total_active_ms` |
| **SQL 参考** | `SELECT SUM(duration_ms) FROM window_sessions WHERE date(start_ms/1000,'unixepoch','localtime') = :date` |
| **展示格式** | `X 小时 Y 分钟` |
| **阈值** | 无分档（纯数值展示） |
| **降级** | `window_sessions` 为空 → 值为 0，报告标注"无活跃数据" |

### A2 · 各 Intent 总时长与占比

| 属性 | 值 |
| --- | --- |
| **ID** | `A2_intent_breakdown` |
| **含义** | 按 `intent` 分类汇总时长 + 计算占比 |
| **存入字段** | `daily_analysis.intent_breakdown`（JSON） |
| **SQL 参考** | `SELECT intent, SUM(duration_ms) AS total_ms FROM window_sessions WHERE date(start_ms/1000,'unixepoch','localtime') = :date GROUP BY intent ORDER BY total_ms DESC` |
| **占比公式** | `占比 = intent_total_ms / A1_total_active_ms * 100`（保留一位小数） |
| **JSON 格式** | `{"编码开发": 7200000, "研究检索": 3600000, "通讯沟通": 1800000, null: 900000}` |
| **展示格式** | 表格：`| 分类 | 时长 | 占比 |`；`intent IS NULL` 显示为"未分类" |
| **降级** | 全部为 `null` intent → 仅显示"未分类"一行 |

### A3 · Top 5 应用时长

| 属性 | 值 |
| --- | --- |
| **ID** | `A3_top_apps` |
| **含义** | 按应用名汇总时长，取前 5 |
| **存入字段** | `daily_analysis.top_apps`（JSON） |
| **SQL 参考** | `SELECT app_name, SUM(duration_ms) AS total_ms FROM window_sessions WHERE date(start_ms/1000,'unixepoch','localtime') = :date GROUP BY app_name ORDER BY total_ms DESC LIMIT 5` |
| **JSON 格式** | `[{"app": "Cursor", "ms": 5400000}, {"app": "Chrome", "ms": 3200000}, ...]` |
| **展示格式** | 列表：`1. Cursor — 1h 30m（18.8%）` |

### A4 · 未分类时长

| 属性 | 值 |
| --- | --- |
| **ID** | `A4_unclassified_ms` |
| **含义** | `intent IS NULL` 的会话总时长 |
| **计算** | 从 A2 的 JSON 中取 `null` 键的值 |
| **展示** | 若占总活跃时长 >20%，在报告总览中提示"XX% 的时间未分类，建议完善 Intent 规则" |

## 三、指标组 B：注意力与切换

**数据源**：`app_switches` + `window_sessions`  
**优先级**：P0  
**里程碑**：M1

### B1 · 每小时切换次数

| 属性 | 值 |
| --- | --- |
| **ID** | `B1_switches_per_hour` |
| **含义** | 按小时桶统计 `app_switches` 条数 |
| **存入字段** | `daily_analysis.switches_per_hour`（JSON）+ `daily_analysis.total_switches`（总数） |
| **SQL 参考** | `SELECT strftime('%H', timestamp_ms/1000, 'unixepoch', 'localtime') AS hour, COUNT(*) AS cnt FROM app_switches WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date GROUP BY hour` |
| **JSON 格式** | `{"09": 12, "10": 8, "11": 23, ...}` |
| **阈值** | 每小时：≤8 → 低（专注）/ 9–20 → 中 / >20 → 高（碎片化） |
| **展示格式** | 文本表格 + 每小时后标注 (低/中/高) |
| **降级** | `app_switches` 不可用 → 跳过本指标，章节标注 |

### B2 · Top 5 切换对

| 属性 | 值 |
| --- | --- |
| **ID** | `B2_top_switch_pairs` |
| **含义** | 最高频的 from_app → to_app 组合 |
| **存入字段** | `daily_analysis.top_switch_pairs`（JSON） |
| **SQL 参考** | `SELECT from_app, to_app, COUNT(*) AS cnt FROM app_switches WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date GROUP BY from_app, to_app ORDER BY cnt DESC LIMIT 5` |
| **JSON 格式** | `[{"from": "Cursor", "to": "Chrome", "count": 23}, ...]` |
| **展示格式** | `Cursor ↔ Chrome（23 次）`（双向合并展示：若 A→B 和 B→A 都在 Top 5，合并为双向） |

### B3 · 深度工作段

| 属性 | 值 |
| --- | --- |
| **ID** | `B3_deep_work_segments` |
| **含义** | 连续 ≥30 分钟无 `app_switch` 且同一 `intent` 的时间段 |
| **存入字段** | `daily_analysis.deep_work_segments`（JSON）+ `daily_analysis.deep_work_total_ms` |
| **算法** | 1. 按时间排序所有 `app_switches`；2. 计算相邻切换的间隔；3. 间隔 ≥30 分钟的时段标记为候选深度段；4. 校验该时段内 `window_sessions` 的 `intent` 是否单一；5. 若 intent 单一 → 确认为深度工作段 |
| **JSON 格式** | `[{"start_ms": 1711872900000, "end_ms": 1711878120000, "duration_ms": 5220000, "intent": "编码开发", "app": "Cursor"}, ...]` |
| **阈值** | ≥30 分钟 = 一个段；段内 intent 变化 → 拆分 |
| **展示格式** | `10:15–11:42（编码开发，87 分钟）` |

### B4 · 碎片化指数

| 属性 | 值 |
| --- | --- |
| **ID** | `B4_fragmentation_pct` |
| **含义** | 高频切换时段在总活跃时段中的占比 |
| **存入字段** | `daily_analysis.fragmentation_pct` |
| **算法** | 1. 将活跃时段划分为连续的 5 分钟窗口；2. 统计每个窗口内的 distinct `to_app` 数量；3. distinct ≥3 的窗口标记为"碎片窗口"；4. `碎片化指数 = 碎片窗口数 / 总窗口数 * 100` |
| **阈值** | ≤15% → 低 / 16–30% → 中 / >30% → 高 |
| **展示格式** | `碎片化指数：23.5%（中）` |
| **边界** | 总活跃时长 <30 分钟 → 不计算，标注"数据不足" |

## 四、指标组 C：打断分析

**数据源**：`notifications` + `app_switches`  
**优先级**：P0  
**里程碑**：M2  
**降级**：`notifications` 表不可用时，仅使用 `app_switches` 中 `switch_type = 'notification'` 的记录，报告标注"仅基于切换推断"

### C1 · 打断者排行

| 属性 | 值 |
| --- | --- |
| **ID** | `C1_top_interrupters` |
| **含义** | 按来源应用统计通知数量 + 导致切换比例 |
| **存入字段** | `daily_analysis.top_interrupters`（JSON）+ `daily_analysis.notification_count` |
| **SQL 参考** | `SELECT source_app, COUNT(*) AS total, SUM(caused_switch) AS switches, ROUND(SUM(caused_switch)*100.0/COUNT(*), 1) AS switch_rate FROM notifications WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date GROUP BY source_app ORDER BY total DESC LIMIT 5` |
| **降级 SQL** | `SELECT to_app AS source_app, COUNT(*) AS total FROM app_switches WHERE switch_type = 'notification' AND date(timestamp_ms/1000,'unixepoch','localtime') = :date GROUP BY to_app ORDER BY total DESC LIMIT 5`（无 switch_rate） |
| **JSON 格式** | `[{"app": "微信", "count": 23, "switch_rate": 65.2, "avg_delay_ms": 4500}, ...]` |
| **展示格式** | `微信 — 23 次通知（65.2% 导致切换）` |

### C2 · 平均响应延迟

| 属性 | 值 |
| --- | --- |
| **ID** | `C2_avg_response_delay` |
| **含义** | 用户响应通知的平均时间 |
| **SQL 参考** | `SELECT AVG(response_delay_ms) FROM notifications WHERE user_responded = 1 AND date(timestamp_ms/1000,'unixepoch','localtime') = :date` |
| **展示格式** | `平均响应延迟：X 分 Y 秒` |
| **降级** | 无 `notifications` 数据 → 不展示 |

### C3 · 打断恢复成本

| 属性 | 值 |
| --- | --- |
| **ID** | `C3_recovery_cost` |
| **含义** | 被通知打断后返回原应用的平均耗时 |
| **算法** | 1. 找到所有 `switch_type = 'notification'` 的切换事件（记为 S1，`from_app` = A）；2. 在 S1 之后的 `app_switches` 中，找到最近一次 `to_app = A` 的事件（记为 S2）；3. `recovery_time = S2.timestamp_ms - S1.timestamp_ms`；4. 取所有配对的 AVG |
| **展示格式** | `平均恢复时长：X 分 Y 秒` |
| **边界** | 若找不到返回事件（用户未返回） → 该次打断不纳入平均 |

### C4 · 深度工作中打断数

| 属性 | 值 |
| --- | --- |
| **ID** | `C4_interrupts_in_deep` |
| **含义** | 深度工作段（B3）内收到的通知或 notification 类切换的数量 |
| **存入字段** | `daily_analysis.interrupts_in_deep` |
| **算法** | 遍历 B3 的每个段 `[start_ms, end_ms]`，统计该区间内的 `notifications` 条数（或降级时统计 `app_switches WHERE switch_type = 'notification'`） |
| **展示格式** | `深度工作中打断：N 次` |

## 五、指标组 D：输入节奏

**数据源**：`input_metrics`  
**优先级**：P0  
**里程碑**：M2  
**降级**：`input_metrics` 表不可用时，整个指标组跳过，章节标注"输入行为数据不可用"

### D1 · KPM 趋势

| 属性 | 值 |
| --- | --- |
| **ID** | `D1_kpm_by_hour` |
| **含义** | 按小时桶统计平均 KPM |
| **存入字段** | `daily_analysis.kpm_by_hour`（JSON）+ `daily_analysis.avg_kpm` |
| **SQL 参考** | `SELECT strftime('%H', timestamp_ms/1000, 'unixepoch', 'localtime') AS hour, ROUND(AVG(kpm), 1) AS avg_kpm FROM input_metrics WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date AND kpm > 0 GROUP BY hour` |
| **JSON 格式** | `{"09": 45.2, "10": 38.1, "11": 52.7, ...}` |
| **展示格式** | 文本表格，标注趋势方向（如"上午高/下午低"） |
| **边界** | `kpm = 0` 的静默行不纳入 AVG（避免 AFK 拉低均值） |

### D2 · 退格率

| 属性 | 值 |
| --- | --- |
| **ID** | `D2_avg_delete_ratio` |
| **含义** | 当天平均退格率 |
| **存入字段** | `daily_analysis.avg_delete_ratio` |
| **SQL 参考** | `SELECT ROUND(AVG(delete_ratio), 3) FROM input_metrics WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date AND keystrokes_count > 0` |
| **阈值** | ≤0.05 → 低（流畅）/ 0.05–0.15 → 中 / >0.15 → 高（犹豫/困难） |
| **展示格式** | `退格率：8.3%（中）` |

### D3 · 心流评分

| 属性 | 值 |
| --- | --- |
| **ID** | `D3_flow_score` |
| **含义** | 综合评估用户进入心流状态的程度 |
| **存入字段** | `daily_analysis.flow_score_avg` |
| **计算单元** | 每条 `input_metrics` 行关联其所在 session 计算一次 |
| **公式** | 见下方 |
| **聚合** | `daily_analysis.flow_score_avg = AVG(所有行的 flow_score)` |
| **范围** | 0–100（整数，四舍五入） |
| **展示格式** | `心流评分：72/100` |

**逐行计算公式**：

```
flow_score(row) = (
    normalize(row.kpm, 0, 120)                             * 0.30
  + (1.0 - normalize(row.delete_ratio, 0, 0.3))           * 0.20
  + normalize(row.typing_burst_count * row.window_interval_secs / max(row.typing_burst_count, 1), 0, 60)  * 0.25
  + normalize(session.duration_ms / 1000, 0, 3600)        * 0.25
) * 100

其中：
- row.kpm：当行的每分钟击键率
- row.delete_ratio：当行的退格率
- burst_duration_avg 近似为 window_interval_secs / max(typing_burst_count, 1)
- session.duration_ms：row.session_id 对应的 window_session 的 duration_ms
```

**权重说明**：

| 因子 | 权重 | 满分线 | 逻辑 |
| --- | --- | --- | --- |
| KPM | 30% | 120 KPM | 高速输入 = 高心流可能 |
| 退格率（反向） | 20% | 0.3 | 低退格 = 思路清晰 |
| Burst 平均时长 | 25% | 60 秒 | 长 burst = 持续输出 |
| Session 时长 | 25% | 3600 秒 | 长时间不切换 = 沉浸 |

### D4 · 挣扎评分

| 属性 | 值 |
| --- | --- |
| **ID** | `D4_struggle_score` |
| **含义** | 综合评估用户遇到困难/卡壳的程度 |
| **存入字段** | `daily_analysis.struggle_score_avg` |
| **范围** | 0–100（整数，四舍五入） |
| **展示格式** | `挣扎评分：35/100` |

**逐行计算公式**：

```
struggle_score(row) = (
    normalize(row.delete_ratio, 0, 0.3)                    * 0.30
  + normalize(row.undo_count / (row.window_interval_secs / 60), 0, 5) * 0.20
  + (1.0 - normalize(burst_duration_avg, 0, 60))           * 0.25
  + normalize(row.longest_pause_ms, 0, 30000)              * 0.25
) * 100
```

**权重说明**：

| 因子 | 权重 | 满分线 | 逻辑 |
| --- | --- | --- | --- |
| 退格率（正向） | 30% | 0.3 | 高退格 = 反复修改 |
| 每分钟撤销次数 | 20% | 5 次/分钟 | 频繁撤销 = 不确定 |
| Burst 时长（反向） | 25% | 60 秒 | 短 burst = 频繁中断思路 |
| 最长停顿 | 25% | 30 秒 | 长时间停顿 = 卡壳 |

### D5 · 心流/挣扎评分互斥性说明

心流和挣扎**不是**同一轴的两端，它们是两个独立维度。一个用户可能：
- 高心流 + 低挣扎 = 状态极佳
- 低心流 + 高挣扎 = 遇到困难
- 低心流 + 低挣扎 = 低强度工作（如阅读、开会）
- 高心流 + 高挣扎 = 罕见，可能是评分校准问题

## 六、指标组 E：信息流向

**数据源**：`clipboard_flows`  
**优先级**：P1  
**里程碑**：P1 池（M1/M2 之后视进度纳入）  
**降级**：`clipboard_flows` 不可用 → 整个章节跳过

### E1 · Top 5 流向边

| 属性 | 值 |
| --- | --- |
| **ID** | `E1_top_flows` |
| **存入字段** | `daily_analysis.top_flows`（JSON） |
| **算法** | 1. 为每个 `flow_pair_id` 找到 action='copy' 的 `app_name`（=来源）和 action='paste' 的 `app_name`（=目标）；2. 按 `(来源, 目标)` 分组计数；3. 取前 5 |
| **SQL 参考** | `SELECT c.app_name AS from_app, p.app_name AS to_app, COUNT(*) AS cnt FROM clipboard_flows c JOIN clipboard_flows p ON c.flow_pair_id = p.flow_pair_id WHERE c.action = 'copy' AND p.action = 'paste' AND date(c.timestamp_ms/1000,'unixepoch','localtime') = :date GROUP BY from_app, to_app ORDER BY cnt DESC LIMIT 5` |
| **展示格式** | `Chrome → Cursor（47 次）` |

### E2 · 总搬运次数

| 属性 | 值 |
| --- | --- |
| **ID** | `E2_clipboard_pairs` |
| **存入字段** | `daily_analysis.clipboard_pairs` |
| **SQL 参考** | `SELECT COUNT(DISTINCT flow_pair_id) FROM clipboard_flows WHERE date(timestamp_ms/1000,'unixepoch','localtime') = :date AND action = 'paste'` |

### E3 · 内容类型分布

| 属性 | 值 |
| --- | --- |
| **ID** | `E3_content_types` |
| **不存入 daily_analysis**（可在查询时按需获取） |

| **展示格式** | `文本 62% · 代码 25% · URL 10% · 图片 3%` |

## 七、指标组 F：环境上下文

**数据源**：`ambient_context`  
**优先级**：P1  
**里程碑**：P1 池  
**降级**：`ambient_context` 不可用 → 整个章节跳过

### F1 · 工作场景识别

| 属性 | 值 |
| --- | --- |
| **ID** | `F1_scene_breakdown` |
| **存入字段** | `daily_analysis.scene_breakdown`（JSON） |
| **推断规则** | 每条 `ambient_context` 行打标签，然后按标签聚合时长 |

**场景推断规则**（按优先级匹配，首个命中即停）：

| 优先级 | 条件 | 标签 |
| --- | --- | --- |
| 1 | `is_camera_active = 1 AND is_audio_input_active = 1` | `会议中` |
| 2 | `is_external_display = 1 AND is_charging = 1` | `办公室` |
| 3 | `is_external_display = 0 AND is_charging = 0` | `移动办公` |
| 4 | `is_dnd_enabled = 1` | `专注模式`（叠加标签，不覆盖场景） |
| 默认 | 以上均不满足 | `常规` |

**时长计算**：每条 `ambient_context` 代表一个 30 秒窗口，场景时长 = `COUNT(标签) * 30000`（毫秒）。

### F2 · 勿扰模式时长

| 属性 | 值 |
| --- | --- |
| **ID** | `F2_dnd_duration` |
| **SQL 参考** | `SELECT COUNT(*) * 30000 FROM ambient_context WHERE is_dnd_enabled = 1 AND date(timestamp_ms/1000,'unixepoch','localtime') = :date` |
| **展示格式** | `勿扰模式时长：X 分钟` |

## 八、衍生指标（跨组合成）

以下指标在报告的"今日总览"章节中使用，由多个指标组合成：

### X1 · 深度工作占比

| 属性 | 值 |
| --- | --- |
| **公式** | `B3_deep_work_total_ms / A1_total_active_ms * 100` |
| **展示** | `深度工作：2h 15m（28.1%）` |

### X2 · 每小时平均切换

| 属性 | 值 |
| --- | --- |
| **公式** | `B1_total_switches / (A1_total_active_ms / 3600000)` |

### X3 · 一句话效率摘要（非 AI 模板）

## 九、阈值校准计划

本字典中的所有阈值（如 KPM 满分线 120、碎片化指数 30%、深度工作 ≥30 分钟等）均为**初始版本**，基于常识和行业参考设定。

**校准流程**：
1. **M1/M2 阶段**：使用真实数据运行分析，收集指标分布
2. **M3 阶段**：检查阈值是否合理（如 90% 的用户都落在"高碎片"区间 → 阈值过低）
3. **调整**：根据 P50/P90 分布重新设定阈值，更新本字典版本号
4. **原则**：阈值调整不应需要修改 `daily_analysis` 表结构，仅影响展示层分档逻辑
